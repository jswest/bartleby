# Bound parse-worker memory so a long ingest doesn't OOM (issue #213)

> Source: [#213](https://github.com/jswest/bartleby/issues/213)

A real 9k-file image/table-heavy ingest ran the machine out of memory ~50 minutes in — RSS climbed steadily, then the OS killed it. The cause was **memory growth, not a too-high static allocation**: the `spawn` pool (`bartleby/ingest/pool.py`, #165) created workers with **no `maxtasksperchild`**, so each worker lived for the whole run and processed thousands of documents without teardown, and docling/torch accrete resident set across many documents (model/layout caches, allocator fragmentation, tensor retention). N long-lived workers each growing → aggregate RSS climbs until OOM. **The fix is recycling, not tuning the estimate** — on a core-rich box the auto-pick binds on the #211 core reservation (e.g. 14 workers on 16 cores), not the RAM cap, so no static per-worker number would have prevented an unbounded-growth OOM at that worker count. **(1) `maxtasksperchild = WORKER_MAX_TASKS` (100).** The pool tears a worker down after that many documents and spawns a fresh one, which re-runs `_init_worker` — including `warmup(config)`, so a recycled worker re-loads the parse models. 100 is tuned to amortize that warmup (a re-load is seconds; at ~tens-of-seconds-per-doc the recycle overhead is ~1–2%) against the memory it reclaims; it's a constant, not a config knob, until a workload shows it needs to vary. **(2) LRU lane reclaim in the progress renderer — forced by (1).** Lanes in `ScribeProgress` (`bartleby/ingest/progress.py`, #170) are keyed by worker identity (`current_process().name`) and freed only at phase boundaries. A recycled Pool worker comes back under a **new** process name, so under recycling the parse phase would accrue more distinct names than lanes, exhaust the free list, and silently drop every recycled worker's progress — the lanes freeze mid-run while the overall bar keeps moving. So `_by_key` became an `OrderedDict` with least-recently-*updated* eviction: each report `move_to_end`s its key, and when no lane is free the oldest key's lane is reused. The evicted key is the dead worker (it stopped reporting when it was recycled), so live workers always show. This only ever triggers in the parse phase — caption/summarize lanes are keyed by stable `threading.get_ident()` thread ids that never recycle. **(3) `PER_WORKER_GB` 2.5 → 4.0.** A secondary, conservative bump: the old 2.5 estimated bare model-load footprint, but image/table-heavy docling parses run well above that under load, so the auto RAM-cap was over-provisioning workers on RAM-rich boxes. 4.0 is an **estimate, not a measurement** — the honest fix is a measured docling RSS, left as follow-up; with recycling now bounding growth, the static number matters less than it did. **Pairs with [#211](https://github.com/jswest/bartleby/issues/211)** (reserve cores): #211 keeps the machine responsive on the CPU axis, this keeps it alive on the RAM axis — together they make auto worker-sizing safe for a multi-thousand-file run. No schema change.
