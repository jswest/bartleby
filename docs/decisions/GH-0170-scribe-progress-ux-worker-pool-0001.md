# Scribe progress UX under the worker pool (issue #170)

> Source: [#170](https://github.com/jswest/bartleby/issues/170)

#165 left ingest with a single relabeled bar + a per-drain stage label, parking the "rich multi-worker run of show" here because the parse-internal page/stage callbacks "can't cheaply cross the queue." This builds it. **The issue predates the omnibus reframe** — it imagined a *flowing* pipeline (parse→caption→summarize→write, with a live count in each) — but #166/#167/#188 landed **three sequential phases** (parse pool of processes → caption thread pool → summarize thread pool, barriers between, each behind the single Writer). So the UX is adapted to that reality, not the flow. **(1) A progress-event channel, main process the sole renderer.** Parse workers are *other processes* and can't draw, so `pool.parse_stream` grew an `on_progress` param: the spawn path stands up a `Manager().Queue()` (proxy-based — it pickles into spawn workers via the initializer where a bare `ctx.Queue()` won't) and a daemon **drain thread** in the main process that pumps `ProgressEvent(worker, item, stage)`s to the renderer while the main thread blocks in `imap_unordered`; a `None` sentinel + `drain.join` + `manager.shutdown()` tear it down in a `finally`. The cheap thing #165 said couldn't cross the queue is exactly this tiny tuple (a few per file), *not* the page-level callback stream. **(2) `parse_fn` grew a `report(stage)` arg** — the pool tags it with the worker (`current_process().name`, or a synthetic `worker-1` inline) and file, so `parse_fn` only names the stage; `_parse_request` composes it with the `--timings` `StageTimer` into the single `on_stage` the parsers already emit (one callback, two consumers), and `report("preparing")` lights a worker's lane the instant it claims a file. Queue errors are swallowed so a progress hiccup can never break a parse. **(3) `bartleby/ingest/progress.py` — `ScribeProgress`, the sole renderer.** A run-of-show header tallying the three phases (`done/total` each, `—` until a phase reveals its denominator, the active phase highlighted); **one overall bar over *every* pipeline unit** (files parsed + images captioned + docs summarized), its total revealed phase by phase so it fills start→finish instead of pinning at 100% after parse; and **one lane per worker** (`● <file> · <stage>`). Built as a `Live(Group(header, overall_progress, lanes_progress))` sharing the `console` module's Console (so `console.warn/error` still insert above the live region, as #165's bar did) — two `Progress` instances (the overall bar carries bar/M-of-N/elapsed; lanes are spinner+text only) rather than one, since lanes aren't a measured quantity. **(4) Lanes are sticky per worker.** A key — a parse process name, or a caption/summarize *thread id* — claims a lane on first sight and keeps it for the phase, its label updating as that worker moves to the next item; phase boundaries free every lane so caption rows don't show leftover parse files. The in-process caption/summarize phases need no queue: `_caption_all`/`_summarize_all` gained an `on_lane(key, item, stage)` callback their workers call directly (keyed by `threading.get_ident()`, so the pool's N threads map to N lanes), additive to the existing `on_progress(done, total)` tally contract — which `_Phase.on_progress` now adapts, so the caller wires `phase.on_progress`/`phase.lane` with no bespoke closures. **`--timings` is unaffected** — it still pins every pool to 1 worker (one lane, sequential), and the renderer is pure display over the same drain. No schema change.
