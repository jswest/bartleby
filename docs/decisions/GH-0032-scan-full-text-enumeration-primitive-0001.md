# `scan` ‚Äî full-text enumeration primitive (issue #32)

> Source: [#32](https://github.com/jswest/bartleby/issues/32)

a new skill script that *filters* rather than *ranks*. Where `search` fuses FTS+semantic via RRF and returns a ranked top-N, `scan` runs FTS5 `MATCH` only and returns *every* matching chunk corpus-wide in `(source_id, chunk_index)` order, paginated with a true `total` so termination is knowable. Documents only (`source_kind='document'`, so `source_id` *is* the `document_id` and grouping needs no join beyond `documents` for `file_name`); summaries/findings/images are never returned. Literal **phrase** match by default (the whole query wrapped as one FTS5 phrase, embedded quotes stripped); `--match-terms` switches to the implicit-AND-of-quoted-tokens behavior `search` uses. Compact by default ‚Äî each hit's `text` is truncated to `--preview` chars (default 240) with `text_length`; full bodies come from `read_chunks --chunks`. `--in-documents` / `--tag` scope it exactly as in `search` (the `_intersect_tag_filter` *shape* is intentionally duplicated across the two scripts rather than hoisted ‚Äî two ~6-line call sites, no speculative shared helper). **Deliberately NOT done: `--offset` on ranked `search`** ‚Äî paginating a fused/semantic RRF ranking has no meaningful termination (you can't distinguish "no more matches" from "scores went low"), so enumeration got its own deterministic filter instead. Rank‚âàfilter only holds for literal phrases on templated corpora (EDGAR/OGE/PACER); on a heterogeneous corpus `scan` degrades to `grep` and `search` is the right tool. Wired into the dispatcher `SCRIPTS` tuple in `bartleby/commands/skill.py`; no schema change.
