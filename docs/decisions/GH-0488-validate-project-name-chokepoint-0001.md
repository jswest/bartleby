# Validate project names at the resolve/open chokepoint (issue #488)

> Source: [#488](https://github.com/jswest/bartleby/issues/488)

`validate_project_name` (a strict anchored regex) was enforced only by `create`/`delete`/`info`/`upgrade`, leaving the hot path open: `resolve_project_name → project_db_path` built `PROJECTS_DIR / name` for any string (`../../foo` included), `set_active_project` checked only `project_dir.exists()` and would persist a traversal name into the active-project config, and the skill runner passed an agent-supplied `--project` straight to `open_db`. An LLM-driven invocation could open any `bartleby.db` on disk or mkdir/write at arbitrary relative paths (session `.active_session` pointers, scribe archive dirs, logs). **The fix validates at the two true chokepoints rather than at every leaf**: `resolve_project_name` (the single resolve-or-fail funnel that `open_db`, the skill runner, scribe, session, and logs all pass through) and `set_active_project` (the only path `project use` takes). One call each covers every code path that turns a name into a filesystem path, and the check runs *before* any path construction, `mkdir`, or `open_db` — so a bad name raises the existing invalid-name `ValueError` with no filesystem touch. **Why not validate inside `project_db_path` / `get_project_dir`?** Those are pure path builders called in contexts (e.g. `list_projects` iterating real on-disk dir names) where re-validating already-trusted names is noise; the chokepoint is the input boundary, not the path constructor. The validation import in `connection.py` is **lazy inside `resolve_project_name`** (alongside the existing lazy `get_active_project` import) because `bartleby.project` imports from `bartleby.db.connection` — a top-level import would be circular. The create-new-project flow is unaffected: `create_project` validates the name itself and never routes through `resolve_project_name`, so creating a *valid* not-yet-existing name still works; validation rejects malformed names, not all nonexistent ones. **Input validation only — no new abstraction, no schema change, no SCHEMA_VERSION bump.** Tests in `tests/test_project.py` assert `resolve_project_name`/`open_db`/`set_active_project` reject `../../x`-style names with the invalid-name error, that no file lands at the traversed path and the active-project config is left unchanged, plus positive controls that a valid name still resolves and a valid existing project still switches.
