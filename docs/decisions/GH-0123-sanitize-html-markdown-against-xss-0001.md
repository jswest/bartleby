# Sanitize `{@html}` markdown against XSS (issue #123)

> Source: [#123](https://github.com/jswest/bartleby/issues/123)

follow-up to #121's iframe sandboxing. The web UI renders `marked.parse(...)` through `{@html}` at two sites — the document **summary** (`documents/[id]/+page.svelte`, model-generated → prompt-injectable) and the **finding body** (`findings/[id]/+page.svelte`, agent-authored) — and `marked` passes raw HTML through, so an embedded `<script>` executed in the app origin (a stored-XSS surface distinct from the source-doc iframe). Both sites flow through the **one shared `marked` singleton** configured in `+layout.svelte` (`marked.use({ hooks: { postprocess } })`, which already rewrote `<a>` to `target=_blank rel=noopener`), so sanitization lives there once rather than per-route — `DOMPurify.sanitize(html)` from **`isomorphic-dompurify`** (runs in both SSR/node via jsdom and the browser, since the finding body is re-parsed client-side to bake the active-chip class). **Sanitize-then-link-rewrite ordering** is deliberate: `target`/`rel` are appended to the already-clean output so they don't depend on the sanitizer preserving them. Relies on **DOMPurify defaults** (no custom allowlist) — empirically verified to strip `<script>`/`on*` handlers/`javascript:` URLs while keeping the cite-chip `<button>` (`type`/`class`/`data-chunk-id`/`title`, the last via the default `ALLOW_DATA_ATTR`); the chips are injected into the markdown *before* parse (`renderChip`) but are our own trusted HTML and survive the allowlist. Verified end-to-end against a throwaway project: a summary and finding body carrying `<script>`/`<img onerror>` render with the dangerous tags stripped and the markdown (and safe links with `target/rel`) intact. No new client/per-route code — `marked.parse` callers are unchanged. No schema change.
