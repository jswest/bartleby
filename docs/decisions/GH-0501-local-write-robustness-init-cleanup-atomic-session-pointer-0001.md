# Local write robustness: failed `init_db` unlinks its partial DB, missing-`meta`-table gets the recreate message, and the active-session pointer is written atomically (issue #501)

> Source: [#501](https://github.com/jswest/bartleby/issues/501)

Three small durability fixes on the local write paths, none touching the schema (`SCHEMA_VERSION` stays 9). **(1)** `init_db` now wraps the DDL-plus-`meta`-populate body in an `except BaseException` that closes the connection, `unlink(missing_ok=True)`s `db_path`, and re-raises. The previous code left a half-built file on disk if anything between `apsw.Connection(...)` and the final `executemany` blew up; the next `create` then hit the `FileExistsError` guard at the top and refused to retry, leaving the user wedged with no clean way forward except a manual `rm`. A partial DB is strictly worse than no DB, so we delete it and let the retry start clean — the connection has to close *before* the unlink or the file and its WAL sidecars stay locked. We catch `BaseException` (not just `Exception`) deliberately: a `KeyboardInterrupt` mid-create is exactly the interruption that strands a partial file, and cleaning up then re-raising preserves the signal. **(2)** `open_db` wraps the `SELECT value FROM meta` in a `try/except apsw.SQLError` that funnels the no-`meta`-table case (a truncated/partial DB, or a foreign SQLite file) into the same friendly "Database at ... has no schema_version. Recreate the project." `RuntimeError` the missing-*row* case already produced — rather than leaking a raw `no such table: meta` SQLError. The two failure shapes are indistinguishable to the user and deserve the same remedy.

**(3)** `write_active_session_id` now writes the id to a sibling `.active_session.tmp` and `os.replace()`s it over the real pointer. `os.replace` is atomic on the same filesystem, so a concurrent `read_active_session_id` (the skill reads this file constantly to learn the current session without opening the DB) sees either the old id or the new one in full, never a truncated line from an interleaved `write_text`. The temp file is a sibling so the rename stays within one filesystem; a `.tmp` left behind by a crash mid-write is inert, because the rename is the only step that publishes a value. The two halves of this issue (init cleanup and the atomic pointer write) are unrelated in mechanism but share one theme — a local write that fails or races shouldn't leave the next caller worse off — so they ship as one change. Tests drive each fix to regression: a create that fails after the DDL runs leaves no file and the retry succeeds; a DB whose `meta` table is dropped yields the recreate message, not a SQLError; and a reader spying via a patched `os.replace` observes the old id right up until the rename lands, with no leftover `.tmp`.
