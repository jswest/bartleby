# Memory-off reads scoped to own session, not blocked (issue #108)

> Source: [#108](https://github.com/jswest/bartleby/issues/108)

a memory-off (`--no-memory`) session could `save_finding` (no guard) but `read_finding`/`list_findings` hard-failed with `MEMORY_OFF` — a write-only state where a run couldn't even read back the finding it just wrote. The point of memory-off is **no-contamination** (don't bias an evaluation run with prior conclusions), and that only requires walling off *other* sessions' findings, not one's own. So the two direct-read commands now scope to the active session instead of refusing outright: `read_finding` fetches the row first (so a bad id is still `FINDING_NOT_FOUND` regardless of memory state) and raises `MEMORY_OFF` only when the finding was authored by a *different* session; `list_findings` adds `WHERE f.session_id = ?` when memory is off, with `total`/pagination reflecting that scoped set. `save_finding` is unchanged (stays open — writing is how a run produces its comparison artifact for the multi-model eval), and **`search --findings` is unchanged**: it still silently drops *all* findings in memory-off, because ranked retrieval surfacing a finding as if it were corpus evidence is the real contamination path (the load-bearing invariant). The old `_common.require_memory_enabled` hard-block (from issue #40) is **deleted**, replaced by a boolean `_common.memory_enabled(conn, session_id)` that both read commands branch on and `search.py` reuses in place of its duplicated inline `SELECT memory_enabled`. The cross-session existence leak (a memory-off caller learns another session's finding *exists* via `MEMORY_OFF` vs `FINDING_NOT_FOUND`) is the same tradeoff #40 already accepted for direct-intent reads. No schema change.
