# Decouple summarization into its own pass (issue #167)

> Source: [#167](https://github.com/jswest/bartleby/issues/167)

under the parse pool (#165) the main process drains each parsed document — persist → caption → **summarize** — inline, one at a time. The #162 benchmark put the per-doc summary at ≈59% of wall-clock, so draining it inline meant the slow LLM call **backpressured the parse workers**: the pool's result queue filled while the main thread blocked on a summary, and the parse cores idled. Lifted summarization out of the per-doc drain into a **separate pass that runs after the parse/caption drain finishes** (`scribe._summarize_documents`). The pass is DB-driven, not stream-driven: `Writer.documents_needing_summary()` returns every document with no `summaries` row and ≥1 summarizable chunk (the same chunk-count guard as `summarizable_chunk_count`, so an image-only doc with nothing extractable is never offered to the model — #80), and the pass summarizes each via the unchanged `_summarize_if_missing` contract (one-shot, truncated to `max_summarize_tokens`). **Consequences:** (1) `_drain` is caption-only and `_is_complete` collapses to `not uncaptioned_images(...)` — the parse/caption drain's resume signal is captions alone; summaries are settled entirely by the pass. (2) **Resume falls out for free, and unifies with fresh work** — the pass picks up whatever the DB lacks, so a corpus parsed with summaries off is fully summarized on a later run with them on, no re-parse, and the old special-casing of the resume bucket for summaries is gone (`_classify` lost its `summaries_enabled` param). (3) `--timings` now merges per-doc records across both passes (`doc_timings` keyed by `document_id`) and emits one coherent line per doc at the end, so the benchmark breakdown stays one-record-per-doc (no double-count). **Stays sequential on the main thread behind the single Writer** — #165's "all concurrency lives upstream in parsing" invariant is preserved verbatim; this is a structural decouple, not added concurrency. The actual summary-throughput win is the provider Batch API (#168), and this pass is exactly the seam it plugs into (collect the work-list → submit as a batch → join at write time). **Deliberately NOT built**: a concurrent summarize pool (would soften the single-writer-on-main invariant before #168 lands — explicitly chosen against); the Batch submission itself (#168); overlapping the summarize pass with the parse drain. **Superseded by #188** — the concurrent summarize pool deferred here was instead built in-process (it keeps the single-writer-on-main invariant, not softens it), and `_summarize_documents` / `_summarize_if_missing` collapsed into `_summarize_all` there; #168 remains the heavier provider-side lever. No schema change.
