# Runner-seam transaction — `run(mutates=True)` (issue #339)

> Source: [#339](https://github.com/jswest/bartleby/issues/339)

Atomicity for mutating skill scripts was per-script and ad hoc: `merge_findings` and `delete_finding` each hand-rolled a `with conn:` block (GH-0056) around their multi-statement writes, while every other write-path script relied on apsw's implicit auto-commit and could leave a half-applied mutation behind if a statement raised mid-way. This moves the rule to **one place — the runner seam** — so atomicity is a property a script *declares*, not boilerplate it remembers. `skill_runner.run` gains a `mutates: bool = False` keyword; when `True`, the runner wraps **only** the `work()` call in a single `with conn:` transaction, so the whole mutation commits or rolls back as a unit. Read-only scripts (the default) open no transaction — behavior identical to before. Three properties are load-bearing and preserved: (1) the transaction wraps `work()` *only*, not project/session resolution — those run before it and commit their own bookkeeping; (2) apsw's `with conn:` is a **deferred** transaction (no write lock until the first write), so a `mutates=True` script that ends up reading-only never grabs a lock; (3) the `with conn:` blocks inside the `bartleby.db.chunks` typed helpers **nest as savepoints** under the outer transaction rather than committing independently, so a partial write (chunks deleted, then the row delete raises) rolls back completely. The audit `log_call` write stays **outside** the transaction on purpose: a mutation that rolls back must still leave an `audit_logs` row recording that it was attempted and failed — the audit log is a record of *calls*, not of *successful* calls. The now-redundant hand-rolled `with conn:` wrappers in `merge_findings` (#56) and `delete_finding` (#56) are deleted; their statements now run unwrapped, exactly as they will once #340 flips them to `mutates=True`. **#339 deliberately flips no real script** — it ships only the seam mechanism, the removal of the two hand-wraps, and the guard test. Consequently `merge_findings`/`delete_finding` are *momentarily un-wrapped* (their multi-statement mutations are no longer atomic) between this PR and #340 marking them mutating; that gap is intentional and closes in #340. The guard test (`tests/test_skill_runner.py`) drives `run` directly with inline `work` callbacks rather than a real script, so the seam is exercised in isolation: a raising `mutates=True` work() is asserted to roll back across `findings`, `chunks`, and the `chunks_fts`/`chunks_vec` mirrors *while* the audit row survives; a `mutates=False` probe is asserted to see `conn.in_transaction is False` (no transaction opened), and a `mutates=True` probe to see it `True`. No schema change (additive bundle — `SCHEMA_VERSION` untouched). **Deliberately NOT done here**: flipping any production script to `mutates=True` (#340), and any change to `tag.py`, which stays unwrapped by design (its per-document commit is load-bearing).
