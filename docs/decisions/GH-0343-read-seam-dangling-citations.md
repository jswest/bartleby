# Surface dangling `[^N]` citation markers at the read seam (issue #343)

> Source: [#343](https://github.com/jswest/bartleby/issues/343)

A finding cites chunks with `[^N]` markers where `N` is the cited chunk's id, and findings may legitimately cite *other* findings' chunks. When such a finding is deleted — or edited/merged, which rebuilds its body chunks under new ids — those chunks disappear and `finding_citations.chunk_id REFERENCES chunks(chunk_id) ON DELETE CASCADE` strips the citation rows pointing at them. The *citing* finding's body is left verbatim, so its `[^N]` marker now points at nothing. That dangling marker is real intel — it records that a claim *was* supported by a source since removed, distinct from a claim never cited — but it was encoded illegibly: the web finding view leaked it as raw `[^123]` text through the chip-render fallback, and `read_finding` returned the same dead marker in the verbatim body. The write paths already reject dangling markers *at save time* (`load_finding_body` → `UNKNOWN_CITATIONS`), so dangling is strictly a **post-write cascade artifact**, which is why the fix lives at the **read seam**, not at any mutation.

Whether a marker dangles is fully derivable from data already in hand: it is any `[^N]` chunk id in the body absent from the finding's *resolved* citations (`resolve_citations` already drops vanished chunks, so the resolved set is the correct thing to diff against). This needs **no stored state, no schema change, and no write-path work**, and computing it at read time covers markers orphaned by *any* cause (delete, edit/merge rebuild, legacy data), not just deletion.

- **`read_finding`** gains a computed `dangling_citations: [int, ...]` field — chunk ids in the body's `[^N]` markers absent from the resolved citations (`extract_citations(body)` minus the resolved chunk-id set; no query beyond what the script already runs). Documented in the `Output:` block and in the SKILL.md `read_finding` entry, with one line telling the agent to **flag** such a marker as a removed citation when compiling a report, not silently drop it.
- **Web finding view** (`+page.svelte`): the marker-render fallback that returned the raw `match` now calls `renderTombstone(N)`, rendering an unresolved `[^N]` as a muted, italic "cited source no longer available" affordance that **keeps the marker present** rather than leaking raw `[^123]`. Reuses the existing `byId` map (this is exactly the existing miss on it) and the existing `esc()` helper; no server / `queries.js` change.
- **`delete_finding` honesty**: the docstring no longer claims a deletion has "no referential fallout for the corpus" — false for finding-to-finding citations, where the cascade severs the *incoming* citation rows of other findings. The docstring now names that fallout and points at the read seam that surfaces it; the `removed_citations` Output-block comment now says explicitly it counts only the deleted finding's **own outgoing** citations. No new count field and no repair worklist.

The state is phrased "**cited source no longer available**," never "deleted": at read time only the id survives, so we cannot distinguish a truly-deleted source from one whose finding was edited/merged so the same content lives on under a new chunk id.

**Deliberately NOT done** (all rejected in the issue): mutating any finding body to annotate dead markers (breaks the verbatim-body contract and triggers another chunk rebuild); a write-time sweep that marks citations dead across the corpus on delete (denormalizes a free-to-derive fact and widens the mutation write-lock window the sibling atomicity work is shrinking); a mutation-time incoming-cascade count in `delete_finding`. **Out of scope**: `search --findings` excerpts (a dead marker inside an excerpt is left as-is), a project-wide "find every finding with dangling citations" audit (its own on-demand query), and the deeper provenance-model question of findings citing volatile finding-kind chunks at all (has a schema dimension). No schema change (`SCHEMA_VERSION` untouched); no re-ingest.
