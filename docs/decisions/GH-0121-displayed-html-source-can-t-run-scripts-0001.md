# Displayed HTML source can't run its scripts (issue #121)

> Source: [#121](https://github.com/jswest/bartleby/issues/121)

the web viewer served `.html`/`.htm` source documents as `text/html` inline (`files/[document_id]/+server.js`) and embedded them in a bare `<iframe src="/files/<id>">`, so an ingested HTML document's `<script>` executed in the app's origin — a stored-XSS vector (source documents are evidence to *read*, never to *run*). Fixed in two complementary layers: (1) the `/files` route adds `X-Content-Type-Options: nosniff` to every response and `Content-Security-Policy: sandbox` to **HTML responses only** — the empty `sandbox` directive neutralizes scripts whether the doc is framed or opened directly in a tab, and HTML is the only script-bearing source type so PDFs/images/text are untouched; (2) both viewers (`documents/[id]`, `findings/[id]`) sandbox the iframe for **non-PDF sources only** (`sandbox={isPdf ? undefined : ""}`, `isPdf` derived from the document / active-citation `file_name`) — PDFs keep the browser's native viewer and `#page=` jumps, everything else renders inert. No `allow-scripts` anywhere; unknown/missing filename → sandboxed (safe default). Verified end-to-end against a throwaway project: an HTML doc's `/files` response carries CSP+nosniff and its `/documents` iframe SSRs `sandbox=""`, while a PDF carries nosniff only and its iframe has no `sandbox`. **Deliberately scoped out (issue #123)**: the `{@html}` markdown render paths (document summary — model-generated, hence prompt-injectable — and finding body) are a *separate* stored-XSS surface, untouched here because sanitizing them must preserve the cite-chip `<button>`s injected through that same `{@html}`. No schema change.
