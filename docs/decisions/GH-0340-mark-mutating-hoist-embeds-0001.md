# Mark write scripts mutating + hoist embeds out of the write lock (issue #340)

> Source: [#340](https://github.com/jswest/bartleby/issues/340)

Building on the runner seam from #339 (`run(mutates=True)` wraps `work()` in one `with conn:`), six write scripts now declare themselves mutating so their multi-statement writes commit or roll back as a unit: `save_finding`, `edit_finding`, `save_summary`, `merge_tags`, `assign_tag`, `unassign_tag`. Before this, each committed its pieces independently (apsw autocommits outside a transaction), so a failure partway through could leave an orphaned finding with no chunks, an edited body with zero chunks and stale citations, a destroyed prior summary with a chunkless replacement, or assignments copied onto a destination tag while the source tag lingered. The flip is a one-keyword change at each `main()`'s `run(...)` call; success behavior and JSON response shapes are unchanged.

**`tag.py` stays deliberately unwrapped** — a comment at its `main()` says why. Unlike the others it loops one LLM classification *per document* (possibly `--all` over the whole corpus) against the user's often-busy local Ollama, tolerates per-document failure by design (the `failed` bucket), and resumes cheaply via the `_already_tagged` skip. One transaction around that `work()` would hold a write lock for minutes-to-hours and roll back every already-classified document on a mid-sweep failure — destroying the resumability that is load-bearing today. Incremental per-document commit is the correct behavior, so `tag.py` keeps autocommitting per document and is **not** `mutates=True`.

**Embedding is hoisted above the first write** in the three scripts that embed (`save_finding`, `edit_finding`, `save_summary`). `embed_texts` is in-process sentence-transformers with a ~5–10s *lazy* model load on the first call in a fresh process (every skill invocation is fresh). With the transaction wrap in place, doing the embed after the first write would hold the write lock across that load, and `busy_timeout` is only 5000ms — shorter than the load — so a concurrent skill call, the web UI, or a concurrent ingest could hit `BusyError`/`INTERNAL_ERROR`. Because apsw transactions are **deferred** (no write lock until the first write), embedding *before* the first write shrinks the lock to the millisecond SQL tail. Mechanically, `rebuild_finding_chunks` in `_common.py` is split into an EMBED phase (`embed_body_chunks(body) -> list[ChunkInput]`, no DB touch — embedding doesn't need the `finding_id`) and a WRITE phase (`write_finding_chunks(conn, finding_id, chunk_inputs)`). `save_finding` and `edit_finding` call `embed_body_chunks` *before* their `INSERT`/`UPDATE` and `write_finding_chunks` at the tail; `save_summary` builds its `ChunkInput`s before the prior-summary delete, then deletes/inserts/chunk-inserts. `rebuild_finding_chunks` survived as a thin composition of the two phases so `merge_findings` (owned by a separate wave and re-flipped to mutating by the stage-manager) could keep importing it unchanged, on the claim that it already held a write txn across embedding by its own design so the hoist would buy it nothing — **that claim was wrong: `merge_findings`' deferred write lock opens only at its `UPDATE findings`, with the embed sitting *after* it, so it had the identical lock-spans-the-model-load exposure; #364 hoists `merge_findings`' embed above the `UPDATE` and retires the now-unused `rebuild_finding_chunks`.**

**Atomicity is pinned by inject-failure tests** — one per wrapped script. Each injects a failure partway through the write and asserts every affected table is exactly as before: no findings/summaries row, no chunks in any of the three chunk tables (`chunks`/`chunks_fts`/`chunks_vec`, verified via `assert_chunk_tables_consistent`), no citation or tag-assignment rows. For `save_summary` the test asserts the prior summary **and** its chunks survive a failed replace fully intact, built on #344's now-chunk-seeded fixture (`summary_a` / `summary_a_chunk_ids`). The injection point matters: the finding/summary tests break `chunks._pack_embedding`, which fires *during* the chunk insert — after the findings/summary row already landed (and, for the summary replace, after the prior summary + chunks were deleted) — so it is a genuine **mid-write** rollback, not a pre-write validation bounce. (A wrong-sized embedding would instead fail in `chunks._validate` *before* any chunk insert, and with the embed now hoisted it would fail before *any* write — a weaker vector for this property.) The three tag tests inject differently: `assign_tag`/`unassign_tag` wrap their per-document `assign`/`unassign` helper to raise on the second document (the first already wrote), asserting neither document's assignment lands; `merge_tags` installs an apsw exec-trace on the runner's connection (via a patched `skill_runner.open_db`) that raises on the source-tag `DELETE FROM tags` — the second write — asserting the copy `INSERT` rolled back too (source tag and its assignments survive, destination carries nothing).

No schema change (additive bundle — `SCHEMA_VERSION` untouched). **Deliberately NOT done here**: flipping `merge_findings`/`delete_finding` to mutating (the stage-manager does that as a separate integration commit; one is being edited by a parallel wave-2 player), and any change to `tag.py`'s autocommit behavior.
