# Leaderboard speed becomes Input tokens/s — the planning number (hotfix to GH-0303)

> Source: hotfix on `main` after GH-0303 landed; no issue.

GH-0303 replaced the leaderboard's speed columns with **Wall s/tok** — load-corrected wall-clock per *output* token. Length-normalized and defined for cloud rows, yes, but normalized by the wrong length: the question the leaderboard exists to answer is *"given a document of x tokens, how long will it take to summarize?"*, and x is the **input** length — what you know up front. Output length is the model's choice, invisible until after the run, so a per-output-token figure can't be multiplied by anything you have. (It also inverted the readable direction: a seconds-per-token figure makes faster models smaller, and ★-frontier reasoning kept flipping signs.)

Resolution: both speed columns (**Tok/s**, **Wall s/tok**) collapse into one — **Input tokens/s** = `prompt_eval_count / (wall_seconds − load_duration_ns/1e9)`, aggregated as the mean of per-doc medians (same methodology). `prompt_eval_count` is recorded for every provider (`usage.prompt_tokens` for cloud), so cloud rows keep a comparable number, with the same footnote caveat: for `†` rows it still counts the datacenter and network. It deliberately charges *generation* time against *input* length — that ratio is exactly "document tokens per second of waiting", the estimator the user wants, not a pure prompt-processing speed.

Knock-ons: the Pareto-frontier speed axis and the quality-tie sort key ride on `input_tps` (cloud rows stay off the frontier via the `ref.local` gate, unchanged); CSV columns `tokens_per_second` + `wall_seconds_per_token` → `input_tokens_per_second`. The live `Tok/s` in the `benchmark run` progress table is untouched — that one shows tokens accruing *now* and was never the leaderboard's number. `tokens_per_second`/`eval_count` stay recorded in the stores (raw telemetry; progress display uses them) — the leaderboard just no longer reads them. Additive, non-schema: `SCHEMA_VERSION` unchanged, no re-ingest, old stores already carry `prompt_eval_count`.
