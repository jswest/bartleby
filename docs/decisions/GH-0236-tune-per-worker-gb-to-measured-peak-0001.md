# PER_WORKER_GB is sized against measured pdfplumber peak RSS, not a model-load estimate.

Supersedes the `PER_WORKER_GB = 4.0` figure from [GH-0213](GH-0213-bound-parse-worker-memory-0001.md) sub-point (3), which set it as an admitted *estimate, not a measurement* and explicitly left "a measured RSS" as follow-up. This is that follow-up; the rest of GH-0213 (worker recycling via `WORKER_MAX_TASKS`, the LRU lane reclaim, reserving cores) stands unchanged.

The auto-sizer in `_resolve_max_workers` picks `min(cpu_count − RESERVED_CORES, free_ram_gb // PER_WORKER_GB)`. The RAM term only constrains the pool when it's smaller than the core term, so on a RAM-rich box a too-low `PER_WORKER_GB` makes the RAM term irrelevant and the core count wins. At `4.0`, a 16-core / 128 GB box launched ~14 parse workers regardless of memory.

Field measurement (2026-06-08, the `ny-puc` `dmm` corpus — 9,648 PDFs up to 58 MB, 128 GB / 16-core box, **pdfplumber** path) sampled live parse workers mid-run: a single worker on a big multi-hundred-page filing reached **10.5 GB** resident, a second **6.3 GB**, the rest ~0.3 GB between documents. The 10.5 GB peak is page renders plus embedded-image extraction — ~2.6× the `4.0` estimate, and well above the ~4 GB docling model-load footprint that #213 had reasoned from (#213 measured docling, not pdfplumber). An earlier run on this corpus drove the machine into swap-thrash and froze it (WindowServer watchdog timeout, ~115 GB in the compressor) — a swap freeze, not a jetsam OOM-kill, because ~14 workers peaking together exceeded physical RAM.

**The decision: `PER_WORKER_GB = 12.0`, sized against the *peak* RSS rather than steady state**, because the freeze is driven by several workers hitting their peak simultaneously. 12.0 is a deliberate over-estimate: on the box that froze it makes the RAM term bind at ~9 workers (`115 // 12`) instead of the core-bound 14, keeping aggregate peak (~9 × 10.5 GB ≈ 94 GB) comfortably under 128 GB. Sizing higher than the issue's suggested 6–8 GB range is intentional — at 8.0 the RAM term still doesn't bind on a 128 GB box (`115 // 8 = 14`), so it would not have shed a single worker on the machine that actually froze. The guiding asymmetry: under-provisioning workers is cheap (a slower ingest), over-provisioning freezes the box.

A single value rather than a per-converter estimate: pdfplumber is the default PDF converter *and* the path that hit the 10.5 GB peak, so it sets the worst case the auto-sizer must survive; a per-converter table would be added complexity for a knob `max_workers` already overrides when an operator knows their corpus is light. No schema change; tuning a runtime constant, no re-ingest.
