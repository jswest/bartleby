# `bartleby ready` skill installer + `ready`→`config` rename (issue #137)

> Source: [#137](https://github.com/jswest/bartleby/issues/137)

getting the skill into `~/.claude/skills/bartleby` was a manual `rm -rf … && cp -r skill …` documented in the README — easy to forget after editing `SKILL.md`, and the `cp` nests a stale copy a level too deep if the `rm` is skipped. Now a first-class command does it. **Three coordinated moves.** (1) **The harness-facing skill relocated `skill/` → `bartleby/skill/` and ships as package-data** (`pyproject.toml`, alongside the `web/**` entries) — it was a repo-root dir absent from the wheel, so a command run after `uv tool install` couldn't find it; bundling it (same reasoning as the `bartleby/web` bundle for `serve`) lets the installer work from an installed tool *and* a checkout. Source is resolved via `Path(bartleby.__file__).parent / "skill"`, not a hardcoded path. (2) **The config wizard `bartleby ready` was hard-renamed to `bartleby config`** (`commands/ready.py` → `commands/config.py`, `tests/test_ready.py` → `tests/test_config.py`, no alias per the no-backwards-compat rule) to free the `ready` verb for "get the skill ready." (3) **The new `bartleby ready`** (`commands/ready.py`, freshly) wipes-then-copies the bundled skill into the dest so the nesting bug is structurally impossible, and reports drift. **"Latest" is a content hash over the skill files, not the version number** — `SKILL.md` is edited constantly off-tag, so two builds at the same `__version__` can carry different skills and a version compare alone would falsely report up-to-date; a hidden `.bartleby-skill` marker (excluded from the hash) carries the version only for the human-readable line. Flags: `--check` (report + exit non-zero on missing/stale, writes nothing — scriptable), `--force` (rewrite when already current), `--dest`. A guard refuses to `rmtree` a non-empty dest that isn't a skill dir (no `SKILL.md`/marker), so a stray `--dest` can't nuke an unrelated directory. README's install/upgrade/Goose sections collapse to `bartleby ready` — notably "Upgrading from a release" no longer needs a separate checkout, since the pinned `git+https` install now carries the skill. No schema change.
