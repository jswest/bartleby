# Schema v8 — single-writer drain + per-unit restartable ingest (issue #164)

> Source: [#164](https://github.com/jswest/bartleby/issues/164)

ingest was parse-and-write interleaved per document, with completeness judged by a coarse `_document_already_ingested` ("any chunk ⇒ done"). That was a *data-loss bug*: a document whose captioning died after its text chunks landed was treated as complete and skipped forever, **permanently losing the missing captions**. Reworked into a producer/writer split that makes ingest resumable by what's *missing*. (1) **One `Writer` (`bartleby/ingest/writer.py`) owns the WAL connection** and is the sole path that persists a unit — parse, caption, or summary — each in its own transaction (all chunk writes still via the `db.chunks` typed helpers). Parse is producer-side and pure (convert + embed, no DB); the per-type `_ingest_*` dispatchers became `_parse_*` returning a `ParsedDocument`. (2) **Atomic per unit, not per document**: `persist_parse` commits the document row + text chunks + *uncaptioned* image rows (`analysis_json IS NULL`) + `document_images` joins as one transaction (never re-parsed); each caption (`persist_caption`, keyed by image byte-hash) and the summary (`persist_summary`) commit independently — a caption failure can't roll back the expensive parse. The VLM call moved *out* of parse: parse archives the prepared JPEG and records the row; the caption stage reads bytes from the archive (so a fresh-process resume works) or an in-process hint (preserving the sparse-page `prefetched_ocr` and sparing a re-read). (3) **Completeness is derived from DB state** — `parsed (documents row exists) ∧ every image row captioned ∧ (summary row exists ∨ summaries disabled ∨ 0 summarizable chunks)` — and `_process_one` runs only the missing stages: a 55-image doc that died at image 30 re-captions 25 and writes the summary; it never re-parses or re-captions finished images. The atomic-parse guarantee means a `documents` row implies a finished parse, so **`_document_already_ingested` and `_cleanup_partial_document` are deleted** (no stranded-partial cleanup — it can't happen). The image-byte-hash dedup is unchanged (a shared image is captioned once; `persist_caption`'s `analysis_json IS NULL` guard makes a re-apply a no-op). (4) **`failed_ingests` (additive table, the schema-v8 bump)** records a deterministically-failing unit (`file_hash`, `file_name`, `stage ∈ {parse,caption,summary}`, `error`, `attempts`, `last_attempt`, PK `(file_hash, stage)`; caption rows key on the *image* hash) so a poison unit is retried up to `MAX_INGEST_ATTEMPTS` (3) then capped — not looped forever — and **surfaced loudly** (end-of-run report + a "Failed units" row in `bartleby project info`), because a silently-skipped unit reading as a green run is the exact trap this fixes. Cleared on success (`clear_failure`). Shipped additive via `_upgrade_v7_to_v8` (just the `CREATE TABLE`) — existing corpora run `bartleby project upgrade`, no re-ingest; the `test_upgrade_chain_walks_*` gate extended to v8. **No functional change on a clean single-producer run** — same end-state; only crash/resume behavior improves. **Prerequisite for the parse pool (#165)**, which will put a real queue + dedicated writer thread in front of the Writer (it must stay the connection's sole owner — apsw connections aren't thread-safe); this PR keeps the producer→Writer handoff synchronous and inline. **Deliberately NOT built**: the queue/writer-thread itself (#165 owns it, and threading a cross-thread error path into `failed_ingests` inside a "no functional change" PR is where the bugs hide), the parse pool, and a `bartleby project failures` listing command (the table is queryable and `project info` surfaces the count).
