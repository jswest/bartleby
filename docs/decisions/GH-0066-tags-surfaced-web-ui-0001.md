# Tags surfaced in the web UI (issue #66)

> Source: [#66](https://github.com/jswest/bartleby/issues/66)

the SvelteKit UI gained tag visibility in three places, all `web/`-only and read-only (no Python, no schema change, `SCHEMA_VERSION` stays 6). (1) **Document chips** — `listDocuments`/`getDocument` (`queries.js`) now attach a `tags` array via a shared `TAGS_JSON_SUBQUERY` (a two-level correlated `json_group_array(json_object(...))` ordered by `name COLLATE NOCASE`, parsed by a `withTags` helper); chips render under the meta line on the list and below the description on the detail page, with the tag's description as a native `title=` tooltip. Untagged documents (the common case) render nothing. (2) **`/tags` index** — a new route backed by `listAllTags()`, which mirrors the `read_tags` skill's SQL: it keeps the `description` and, unlike the pre-existing `listTags()` (kept as-is for the search filter, where empties are noise), retains zero-document tags via `LEFT JOIN` — a vocabulary entry is worth showing before anything is assigned. (3) **`/tags/[id]` detail** — `getTag()` returns the tag plus its documents in the exact `listDocuments` shape; the route reuses a new shared `DocumentList.svelte` component (extracted from the documents list, so chips render identically in both places). Chose the `/tags/[id]` shape over a `/documents?tag=` filtered view because it is self-contained and the nav's existing `isActive()` child-path match lights the Tags link for free. The `stripExt` title helper moved to `web/src/lib/format.js` (shared by the list component and the detail page).
