# `--help` prints the response shape, not just flags (issue #47)

> Source: [#47](https://github.com/jswest/bartleby/issues/47)

every skill script already documented its JSON return contract in its module docstring's `Output:` block, but the agent never saw it — it reads SKILL.md and `--help`, and `--help` showed arguments only. The contract was half-invisible at the moment it was needed (observed: an agent assumed `scan`'s hits were under `results` like `search`, hit `KeyError`, recovered by dumping keys). Fix: a shared `build_arg_parser(prog, doc)` in `skill_runner.py` that sets the module docstring as the parser `description` with `RawDescriptionHelpFormatter` (so the `Output:` JSON block isn't reflowed); all ~18 scripts construct their parser through it passing `__doc__`. The docstrings stay the single source of truth — no second contract to drift, which is exactly why the "Returns: block in SKILL.md" option was rejected. The raw-formatter+description pairing is a correctness invariant (description without the raw formatter mangles the JSON), so it's centralized in the helper rather than copied to each call site. SKILL.md now tells the agent to run `--help` before parsing a script's output for the first time and flags that top-level keys differ by script on purpose (`search`→`results`, `scan`→`matches`, `list_documents`→`documents`, `list_findings`→`findings`). The `--schema` flag option was rejected as the most code for no more benefit — it would invent a new machine-readable schema per script as a second source of truth when the docstrings already exist.
