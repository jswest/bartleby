# Tesseract owns image transcription, VLM owns description.

Each embedded image is dispositioned by Tesseract first: if it yields ≥ 300 chars at avg confidence ≥ 60, classification is `kind='text'` and the VLM is never called — Tesseract output becomes the single `image_ocr` chunk. Otherwise classification is `kind='scene'` and the VLM produces a bounded description (~200 words, never asked to transcribe) which becomes the single `image_description` chunk. One chunk per image, not two. The split exists because asking a small local VLM (gemma3, etc.) to verbatim-transcribe a text-heavy slide under JSON-grammar constraint causes per-image latency in the tens of minutes; Tesseract handles it in a second. Sparse-page renders that already failed page-level OCR pass their `OcrResult` down so the image pipeline doesn't re-OCR the same bytes. Providers' `analyze_image` returns a `VlmDescription` (description + notes only); the orchestrator merges it with the Tesseract result into the persisted `ImageAnalysis`. **OCR is only the classifier, not the captioner** — so if Tesseract *crashes* (`analyze` catches any exception from `ocr_module.run`), the image degrades to `kind='scene'` and routes to the VLM rather than being dropped (issue #181, decision-log entry below).
