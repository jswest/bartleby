# Opt-in per-stage ingest timing + benchmark mode (issue #162)

> Source: [#162](https://github.com/jswest/bartleby/issues/162)

the concurrency work that follows needs "pages/sec at N workers" and a per-stage breakdown to validate against, and one baseline question settled first — is per-doc time dominated by parse, or by the ~55 captions/doc? Added a `--timings` flag to `bartleby scribe`; **off by default the ingest path is byte-for-byte unchanged** (no `StageTimer` constructed, nothing printed to stdout). **The instrumentation reuses the existing `on_stage(label)` callback** rather than threading a new timing param through every `_ingest_*` function — that callback already fires at each phase boundary in all paths, so a `StageTimer` driven off it (in `bartleby/lib/timing.py`) gets accurate per-stage wall-clock for free; the alternative (a parallel timing param) would duplicate the same threading the callback already does. The raw labels are mapped to a canonical vocabulary (`extracting`→`parse`, `embedding`→`embed`, `analyzing image(s)`→`caption`, `summarizing`→`summarize`); time before the first mark (hash + archive copy) lands in `prep`. **Chunk writes fold into `embed`** — the `INSERT` immediately follows `embed_texts` under the same label, and the baseline question doesn't need them split, so the deliberately-coarse seam was kept rather than instrumenting the inner functions. Per-doc lines (total + split) go to **stderr**; the aggregate — `docs`, `pages`, `wall_clock_s`, `docs_per_s`, `pages_per_s`, and per-stage `{total_s, pct, mean_s}` — is one JSON object on **stdout** (the shared console is stderr-bound, so `--timings > bench.json` captures cleanly). **Rates use whole-run wall-clock** measured around the loop, not the sum of per-doc totals, so inter-doc overhead counts and runs stay comparable. **A doc that raises mid-process (e.g. a summarizer failure) is not timed** — it's caught by the loop's `except` before the record is appended — and **already-ingested files are skipped**, so a clean baseline means a fresh project. `pct` is the share of summed stage time (the parse-vs-caption answer); the empirical number itself is left to the operator's representative sample. No schema change.
