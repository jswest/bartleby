# `scan --sort {document,date}` ‚Äî chronological enumeration (issue #118)

> Source: [#118](https://github.com/jswest/bartleby/issues/118)

`scan` enumerated in a fixed `(document_id, chunk_index)` order; the one analyst move that order can't serve is "walk every match chronologically" on a templated corpus (every filing mentioning X, oldest‚Üínewest). Added a single-valued-now enum `--sort` (matching `--count-by`'s room-to-grow shape). `document` (default) **is** the prior order ‚Äî output stays byte-identical, no compat concern. `date` is a **document-level reorder**: it sorts the doc *groups* by `authored_date` while keeping chunks positional within each doc (`(s.authored_date IS NULL), s.authored_date, c.source_id, c.chunk_index`), so the document-grouping contract and `total`-based termination survive. **Deliberately oldest-first ‚Äî the inverse of `list_documents --sort date`'s newest-first** ‚Äî because scan's use case is reading a corpus forward in time, not triaging the latest; the divergence is intentional and documented in both `--help` surfaces. `authored_date` lives on `summaries` (schema v5) and is NULL-heavy (summarizer-inferred), so both queries gained a `LEFT JOIN summaries s ON s.document_id = c.source_id` (explicit `ON`, not `list_documents`' `USING` ‚Äî scan's left key is `chunks.source_id`, not `document_id`) and a leading `(authored_date IS NULL)` term floats undated docs last, with `(document_id, chunk_index)` as the deterministic tiebreaker so pagination stays stable. `--sort` applies to `--count-by document` too (the issue left this optional ‚Äî taken for consistency): `date` reorders the histogram chronologically instead of by hit count. Two ORDER BY maps (`_CHUNK_ORDER_BY`, `_DOC_ORDER_BY`) rather than one because the per-chunk and histogram queries have genuinely different `document` orders (`source_id, chunk_index` vs `n DESC, source_id`) and tiebreakers ‚Äî collapsing them would need conditional suffixing, more complex not less. The `scan` envelope does **not** echo `sort` (matching `list_documents`, which doesn't either). No schema change.
