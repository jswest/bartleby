# `merge_tags` assignment count — `inserted` + `already_present` (issue #341)

> Source: [#341](https://github.com/jswest/bartleby/issues/341)

`merge_tags` copies a source tag's assignments onto a destination tag with `INSERT OR IGNORE INTO document_tags ... SELECT ... WHERE tag_id = src` and then deletes the source tag. It used to report a single `moved_assignments` field taken from the source tag's pre-merge `COUNT(*)`. That count is a lie whenever the two tags overlap: `INSERT OR IGNORE` silently drops any (document_id, tag_id) the destination already held, so a merge of two tags that both cover the same documents reported every row as "moved" when zero rows were actually written. The over-report is invisible to the agent reading the JSON — there is no way to tell a clean move from a no-op absorb.

The fix splits the field in two rather than renaming it (per the run-wide director's note): `inserted` is the true number of rows written, read from `conn.changes()` immediately after the `INSERT OR IGNORE` (apsw's per-connection rows-changed counter, already used this way in `ingest/writer.py`); `already_present` is `pre_merge_count - inserted`, i.e. the source assignments the destination already carried. Their sum is still the source tag's pre-merge assignment count, so no information is lost — the ambiguous number is decomposed into the two facts an agent curating tags actually wants. Reporting both (not just `inserted`) keeps the merge auditable: "I merged a 40-doc tag into b; 12 were already there, 28 are new" is a materially different outcome from "28 brand-new assignments," and the agent should be able to see which happened.

`conn.changes()` is read *before* the `DELETE FROM tags` so the count reflects only the copy. No `with conn:` transaction is added here — copy+delete atomicity is the runner-seam wrap landing in sibling issue #340; this change is purely the count semantics. No schema change (`document_tags` is untouched), so `SCHEMA_VERSION` is unaffected and no re-ingest is needed. `moved_assignments` had no consumers outside the script itself (grep across the repo found only `merge_tags.py`; `SKILL.md` describes `merge_tags` but never named the field), so the rename is contained to the script's docstring + return and the tag tests.
