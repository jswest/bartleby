# Untrusted `.md` source renders in a sandboxed `srcdoc` iframe with a `default-src 'none'` CSP.

The two-pane detail layout (`/findings/[id]`, `/documents/[id]`) showed `.md` source documents as raw text — the `/files` endpoint serves markdown as `text/plain` and the viewer dropped it into a bare `<iframe src>`, so a reader saw literal `#`/`**`/`|` markup instead of rendered prose. The fix renders the markdown, but the *how* is the load-bearing call: source documents are **untrusted ingested content** (anything pulled from the web, a third party, or decant), unlike finding bodies and summaries, which the agent authors. Two threats had to be handled together, and they are handled by **different** controls. (1) **Script injection** — markdown is a superset of HTML and `marked` passes raw HTML through; this was *already* covered repo-wide by the global `marked.use` DOMPurify postprocess hook in `+layout.svelte`, which strips `<script>` and `onerror`/`onload` handlers before any `{@html}`. (2) **IP/network beaconing** — even with scripts blocked, a passive `<img src="https://tracker/pixel.png">` (trivially produced by markdown `![](…)`) fires an outbound request on view, leaking the host's IP and the fact/time of viewing. DOMPurify keeps remote images by default, and an iframe `sandbox=""` does **not** block subresource loads — so neither of the script-focused controls stops the beacon. The **only** control that does is a `Content-Security-Policy: default-src 'none'`, which prevents the fetch from ever leaving the machine. So the viewer renders `marked.parse(source)` (still flowing through the global DOMPurify hook, as defense in depth) into an iframe `srcdoc` carrying `<meta http-equiv="Content-Security-Policy" content="default-src 'none'; style-src 'unsafe-inline'">` and `sandbox=""`: rendered markdown, scripts structurally impossible, zero outbound requests. This also closed a **pre-existing latent leak** — `documents/[id]` already rendered source markdown inline via `{@html}` with no CSP, so it would have beaconed; both panes now route through one shared `SourceViewer` component (markdown via the sandboxed `srcdoc`; PDF via the native viewer; everything else a sandboxed raw-file iframe), collapsing the previously-duplicated "sandbox-everything-except-PDF" logic. Rejected: **inline `{@html}` rendering** (the existing documents pattern — looks native and reuses `.markdown-body`, but carries the beacon and rests entirely on DOMPurify staying bug-free); **adding DOMPurify image/remote-stripping to the global hook** (broad blast radius across summaries/findings, and still weaker than CSP — it strips the tag rather than guaranteeing no request). The iframe can't reach `app.css` or its CSS variables, so a compact neutral stylesheet is inlined in the `srcdoc` (`style-src 'unsafe-inline'` exists only to enable it). Frontend-only; no `SCHEMA_VERSION` bump (issue #567).
