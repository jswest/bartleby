# Parallel parse pool (issue #165)

> Source: [#165](https://github.com/jswest/bartleby/issues/165)

ingest parsed one document at a time on one core — docling layout models, OCR, embedding, all serial — so a 9k-doc corpus took days. Parsing one document never needs another, so it parallelizes cleanly. #164's producer/writer split set this up: `_parse_document` is already pure (no DB) and the `Writer` already owns the connection. **(1) One pipeline, one swappable scheduler.** `bartleby/ingest/pool.py` is a *generic* executor — `parse_stream(requests, parse_fn, config, max_workers, …)` — that knows how to *schedule* a parse function, not how to parse. It imports nothing from `commands/` (no layering inversion): the caller passes a picklable `parse_fn(request, config)` and `config`, which travel to workers as data. At `max_workers <= 1` requests run inline in-process; above, across a `spawn` pool. **Both paths call the same `parse_fn` and yield the same outcomes** — there is one parse pipeline, the scheduler is the only fork (~8 lines). The inline path is the honest degenerate case (spawning a pool of 1 to work serially is pointless) and the path the mocked test suite drives, since monkeypatches don't cross `spawn`. **(2) Workers parse, the main process drains.** `scribe._parse_request` (the `parse_fn`) archives + parses one file into a `ParsedDocument` and never raises — a parse failure rides back as a `ParseOutcome.error` so one bad file can't tear down the stream. The main loop classifies sources up front (`_classify`: hash + resume lookup → skip / resume / parse), then drains the parse stream through the single Writer one result at a time (`imap_unordered`, completion order), captioning + summarizing each inline. No writer *thread* — the main thread is the drain — so #164's predicted "dedicated writer thread" never materialized; the Writer stays the connection's sole owner on one thread because the concurrency is entirely upstream in parsing. **(3) Embeddings run in-worker** (resolving #165's open question): `embed_texts` already lives inside the `_parse_*` functions, so each worker holds one `SentenceTransformer` and embedding parallelizes with parse for free. The alternative (a batched MPS embed stage on the main process) is a possible later optimization, not built. Both heavy models (`docling._converter`, `embed._model`) are `@lru_cache` → once-per-process regardless; the pool initializer calls new public `prewarm()` seams so the load is paid at worker startup, not on the first document. **`spawn`, not `fork`** is mandatory — the ML runtimes load native threads unsafe to inherit across a macOS fork; docling stays CPU (its layout models crash on MPS). **(4) `max_workers` is config-only** (honoring "new knobs live in `bartleby config`, no CLI flags"; resolving #165's other open question against a per-run override). Unset → auto-pick `min(cpu_count, free_ram_gb // PER_WORKER_GB=2.5)`, floored at 1, so a CPU-rich/RAM-poor box doesn't launch more parse workers than memory holds and OOM; an explicit value over the auto-pick is honored with a warning. New dep `psutil` for the free-RAM read. **(5) The in-process caption hints are deleted.** #164 carried the scaled `jpeg_bytes` + sparse-page `prefetched_ocr` on `ParsedImage` so same-process captioning skipped a re-read/re-OCR; across a pool queue those are dead weight (megabytes/doc, plus pickling an `OcrResult`). Dropped from `ParsedImage`, `_caption_image`, `_caption_missing`, and swept out of `PdfPage.ocr_result` / `images.analyze(prefetched_ocr=…)` — the caption stage now *always* reloads the prepared JPEG from the archive and re-OCRs (the "fresh-process resume" path that already existed). One caption path, which is what lets a parsed result cross the queue cheaply. **`--timings` (#162) pins `max_workers=1`** — its per-stage breakdown is a sequential baseline, meaningless once docs parse concurrently and overlap; the parse-side stages are captured by a `StageTimer` built inside `_parse_request` (in whatever process runs it) and returned as data, so #162's instrumentation survives intact on the inline path. **Deliberately NOT built**: the rich multi-worker progress UX (#170 owns it — under the pool the parse-internal page/stage callbacks can't cheaply cross the queue, so this PR keeps an overall files bar + a per-drain stage label and leaves the per-worker "run of show" to #170); the provider Batch API (#168). No schema change.
