# No summary for zero-chunk documents + image-only summary input (issue #80)

> Source: [#80](https://github.com/jswest/bartleby/issues/80)

the summarizer confabulated complete, plausible, *entirely fabricated* summaries for documents with no extractable content. An image-only / sub-threshold PDF (e.g. an annotated track-map scan) yields a `full_text` of a few trace tokens — page labels, empty page strings joined by `"\n\f\n"` form-feeds — while creating **0 chunks**. That trace is non-empty after `.strip()`, so the old whitespace-only guards (`_maybe_summarize`, `summarize`) waved it through, and the structured-output prompt (no "insufficient content" branch) pressured the model to fill every field → it invented a generic business doc. The summaries were indistinguishable from real ones in `list_documents`/`describe_corpus` and indexed/searchable (`search --summaries`), quietly poisoning triage. **Two coupled fixes**: (1) `_maybe_summarize` now gates on a real chunk count via `_document_chunk_count` (document chunks `source_kind='document'` **plus** image chunks joined through `document_images`) — **0 chunks → skip**, with a stderr `console.warn` so the user can spot the ingestion failure (image-only PDF needing OCR/re-ingest) instead of it masquerading as success. The old `not full_text or not full_text.strip()` check is gone — chunk count subsumes it and correctly admits image-only docs whose `full_text` is empty but which *do* have image chunks. (2) `_build_summary_input` no longer short-circuits to `full_text` when a doc has image chunks but no document chunks (the old `if not doc_rows: return fallback_text` discarded the VLM/Tesseract descriptions and fed the model the same trace garbage — the more common path once a vision provider is configured). It now interleaves whatever chunks exist (document, image, or both); `fallback_text` is reachable only with zero chunks, which (1) already excludes. With (2) in place, `_ingest_image_file`'s post-hoc `full_text` reconstruction from chunks became dead and was deleted (it returns `""`). No write-path validation added on `_write_summary` and no `list_documents`/`describe_corpus` "low extracted content" surfacing — the issue flagged both as optional polish, and skipping at the source means nothing malformed is written to validate. No schema change (`SCHEMA_VERSION` stays 7).
