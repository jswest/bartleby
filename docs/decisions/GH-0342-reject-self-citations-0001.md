# Reject citations to chunks a merge/edit is about to destroy (issue #342)

> Source: [#342](https://github.com/jswest/bartleby/issues/342)

Findings may legitimately cite *finding-kind* chunks (the agent sees those ids via `search --findings`), and `validate_chunk_ids_exist` only checks "this chunk exists right now" — it doesn't restrict `source_kind`. But `merge_findings` and `edit_finding` delete-and-rebuild the body chunks of the findings *involved* in the operation, and `finding_citations.chunk_id REFERENCES chunks(chunk_id) ON DELETE CASCADE` (with FKs on) turns a citation to one of those chunks into one of two silent failures: (a) a citation row inserted while the chunk still exists gets **cascade-deleted** when the owning source is consumed, leaving a `[^N]` marker in the surviving body with no citation row; or (b) the chunk is **deleted before** `replace_finding_citations` runs (the target in a merge, the finding itself in an edit), tripping an immediate FK violation that surfaces as an opaque `INTERNAL_ERROR`. The sibling read-seam work (#343) only *surfaces* dangling markers in *other* findings post-hoc; this is the complementary involved-finding half — refuse upfront so the agent gets an actionable message naming exactly which markers to fix.

The fix is one upfront ownership check, `reject_citations_to_involved_findings` in `_common.py`, run after `load_finding_body` (citations already extracted + existence-validated) and after the existing not-found / memory-wall gates, but **before any write**. It queries the cited *chunk rows* — `source_kind = 'finding'` AND `source_id IN (<involved finding ids>)` AND `chunk_id IN (<citations>)` — and raises `SkillError(code, …, offending_chunk_ids=[…])` if any match. Ownership is decided by the chunk row's `source_id`, never by matching body text (the issue is explicit about this: text matching would both miss renumbered ids and false-positive on prose). One check covers both failure modes because both stem from the same fact — the cited chunk belongs to a finding whose chunks won't survive — so there's no need to distinguish "cascade" from "FK violation" at the call site.

- **`merge_findings`**: involved set is `sources + [target]` (the same `ids` list the not-found / accessibility gates already use). Code `CITES_MERGED_CHUNKS`. Covers both modes the issue documents (citing a `--from` source → silent cascade; citing the target's pre-merge chunks → FK `INTERNAL_ERROR`). No transaction wrap was added and `mutates` is left unset — per #339, the stage-manager flips `merge_findings` to `mutates=True` as a separate integration commit; this issue touches only the citation-rejection check.
- **`edit_finding`**: involved set is `[finding_id]`; check runs only when `--body-file` is supplied (a title/description-only edit rebuilds no chunks). Code `CITES_OWN_CHUNKS`. `edit_finding` is already `mutates=True` (#340), so the check runs inside the runner's transaction — but it raises before the first write either way.

The two codes are distinct only for message ergonomics (`merge` vs. `edit` wording, and which findings are named); they share one helper and one `offending_chunk_ids` field. Tests: `test_skill_merge_findings.py` parametrizes the reject over both a `--from` source and the target (asserting `CITES_MERGED_CHUNKS`, the offending id, and that nothing was deleted); `test_skill_edit_finding.py` adds a self-citation reject (`CITES_OWN_CHUNKS`, finding left intact). The pre-existing happy-path tests — which cite *document* chunks, the normal case — still pass, confirming the check is scoped to finding-kind chunks owned by involved findings and doesn't touch ordinary citations. No schema change (`SCHEMA_VERSION` untouched); no re-ingest.

**Deliberately NOT done**: any change to how non-involved finding-kind citations are handled (those are the #343 read-seam's job, and a still-valid cross-finding citation to an *uninvolved* finding is left untouched); any transaction or `mutates` change to `merge_findings` (owned by the stage-manager per #339); softening to a warning rather than a hard reject (the issue calls for refusal — a `[^N]` with no citation row is corrupt intel, not a soft problem).
