# Web view delegates read parity to the skills — home + `/documents` (issue #110)

> Source: [#110](https://github.com/jswest/bartleby/issues/110)

the web view (`bartleby serve`) lagged the CLI's read surface — the home page was a bare doc/finding count and `/documents` dumped every row, unfiltered and unpaginated. The fix brings the two highest-value gaps to parity **by delegating to the skill scripts via `runSkill`, not re-deriving their SQL in JS** — the same seam `/search` already uses. The dividing principle (settled after weighing "always delegate"): **delegate where the skill owns semantics we'd otherwise duplicate** (ranking, filtering, date/null handling, polymorphic-chunk logic), **keep direct SQL for cheap display reads and the enrichment layer the skills can't give us**. Not "always delegate" — a Python subprocess per request buys nothing on a trivial read, and the skills emit *agent-shaped* JSON so a JS enrichment/SQL layer survives regardless (`enrichHits` is already subprocess + `titleMeta` SQL). `/` now renders `describe_corpus` (token/chunk totals, authored-date range **always paired with the undated count** so a range can't overstate coverage, year histogram, summary coverage, content mix, tag chips, largest docs), falling back to nav cards on skill error. `/documents` renders `list_documents` with authored-date filters (+ include-undated toggle, surfacing `excluded_null_dated`), tag filters, pagination, and sort. Two display gaps the agent-shaped skill doesn't cover, handled web-side: (1) **tags** — the skill returns none, so a batched `tagsByDocument()` query re-attaches chips (the `titleMeta` pattern); the now-unused `listDocuments()` SQL was deleted (no-dormant-code). (2) **sort** — the skill ordered only by `document_id` and sort must precede `LIMIT/OFFSET` (can't be fixed in JS under pagination), so a new **`--sort {id,title,date}`** flag was added to `list_documents.py` (default `id` preserves the agent-facing ingest order; web passes `title`) — a small, agent-useful addition to the skill's contract, every option breaking ties on `document_id` for stable paging. Shared `toSkillError()` (error-envelope normalization) and `clampInt()` (query-param parsing) were hoisted out of the search loader into `skill.js` / `params.js` rather than copied. The web DB stays **read-only by design** — no mutations, no session/project management. **Deliberately deferred** (parity backlog, not this issue): `read_document --full` extracted-text view and `read_chunks` chunk browse (P2); a `logs` audit-log viewer and `project info` panel (P3); any write path (findings/tags/summaries mutation, LLM `tag`), which would require dropping read-only mode and a web write/audit story. No schema change.
