# Semi-automated releases — tags as version pins, minor coupled to `SCHEMA_VERSION` (issue #100)

> Source: [#100](https://github.com/jswest/bartleby/issues/100)

real users installing from a git checkout had no way to pin a known-good version and no signal for the one update that actually hurts — a pull that bumps `SCHEMA_VERSION` and forces a re-ingest. Releases are now git tags of the form **`v0.<SCHEMA_VERSION>.<patch>`**: the **minor *is* the schema version** (not merely bumped alongside it), so the version number self-documents the schema (`v0.8.3` = "schema 8, third patch") and a user can read `SCHEMA_VERSION` out of their DB's `meta` and know which minor line their corpus belongs to. Patch increments per release at the same schema and resets on a bump; major stays a manual human call. Three pieces: (1) **`setuptools-scm`** derives the package version from the latest tag (`dynamic = ["version"]`, static `version`/`__version__` deleted; `__init__.__version__` now reads `importlib.metadata`), so the release script's *only* write is `git tag` — no two-places-to-drift. Off a tag, dev builds report `0.7.0.devN+g<sha>`, surfaced by the new `bartleby --version`. (2) **`scripts/release.py`** (a standalone maintainer chore, deliberately **not** a `bartleby` subcommand — it stays off the end-user CLI surface) computes the next version with no diffing, since the constant *is* the minor: read `SCHEMA_VERSION`, parse the last `v0.*` tag, bump. It tags, and on `--push` publishes a GitHub Release with notes auto-generated from `git log` (a re-ingest banner prepended on a minor bump). (3) A **drift guard** in the same script diffs `schema.py`'s DDL (via AST, not regex) against the last tag and **refuses to tag** when the DDL changed but `SCHEMA_VERSION` did not — the one mistake the version arithmetic can't see (a breaking change mislabeled as a safe patch). The pure functions (version arithmetic, schema parse, drift, notes) are unit-tested; git/gh side effects are not. Baseline tag is **`v0.7.0`** (schema is 7), cut from `main` post-merge by running the script (dogfooding it). **Deliberately NOT built**: a hand-maintained `CHANGELOG.md` (auto-generated GH Release notes replace it — zero upkeep) and PyPI publishing (the unreachable WSJ-internal `wsjpt` git dep and the `--with` install dance make PyPI awkward, and git-tag installs already give real pins). No schema change.
