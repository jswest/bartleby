# Aggregation/facet тАФ `scan --count-by document` + filtered `describe_corpus` (issue #104)

> Source: [#104](https://github.com/jswest/bartleby/issues/104)

two decisive "quantify a pattern" needs an agent kept rolling Python over the existing primitives тАФ (A) "how many *documents* match this marker?" (`scan` gave a chunk `total`, then `len(set(document_id))` by hand) and (B) "what's the content/year/tag mix of *this slice*?" (`describe_corpus` faceted beautifully but corpus-wide only). Solved as **extensions, not a new verb**. (1) `scan --count-by document` (single-valued enum, room to grow) flips to aggregate mode: instead of `matches` it returns a per-document hit histogram (`documents: [{document_id, file_name, chunk_count}]`, `ORDER BY chunk_count DESC, document_id`, paginated by document) plus two always-full rollups тАФ `distinct_document_count` (**the headline** the agent reports in prose) and `total_chunk_count` (the old `total`). Both present so the agent never guesses which `total` meant. Conflicts with `--preview`/`--brief` (they only shape per-chunk matches) тЖТ argparse `p.error` exit-2 (chosen over silent-ignore). (2) `describe_corpus` gained the same scope flags `search`/`scan`/`list_documents` take (`--tag`, `--in-documents`, date bounds) and recomputes *every existing* facet over the slice тАФ **not** a general GROUP-BY engine, the four dimensions are fixed. The filtered `tags` facet lists **only tags present in the slice** (INNER JOIN over slice docs) vs the corpus-wide LEFT-JOIN-all-tags; `content_mix` reaches image chunks through `document_images` since they key on `image_id`, not `document_id`; `document_count`/`token_count` come from the `documents` table (not `len(scope)`) so unresolvable `--in-documents` ids can't inflate the count; an empty slice short-circuits to an all-zero overview. (3) **Refactor тАФ `_tags.resolve_scope`**: the date-filter logic (`--authored-after`/`-before`/`--include-nulls` validation + the `IS NULL`/`excluded_null_dated` bookkeeping) was inline in `list_documents`; hoisted into a `Scope` dataclass + `resolve_scope` that folds tag тИй in_documents тИй date into one resolved `document_ids` set (None = whole corpus, `[]` = matched-nothing short-circuit) and computes `excluded_null_dated`. All three scripts call it; `list_documents` switches to the resolved-id path with **byte-identical output** (the issue mandated no behavior change). This **reverses the #32 "don't hoist `intersect_tag_filter`, two call sites isn't worth a helper" stance** тАФ now three consumers plus a date dimension justify it (`resolve_scope` wraps `intersect_tag_filter`, leaving `search`'s use of the latter untouched). **Echo consistency**: a `filters` *object* (`{tags, in_documents, authored_after, authored_before, include_nulls, excluded_null_dated}`) appears on `scan` and `describe_corpus` only when a filter is active тАФ nested rather than top-level because `describe_corpus`'s top-level `tags` key is already the distribution facet (a flat `tags` echo would collide), and chosen across both for cross-command consistency. `scan`'s old always-present top-level `in_documents`/`tags` moved *into* `filters` (no-backwards-compat rule). `list_documents` keeps its top-level `excluded_null_dated` тАФ the one accepted inconsistency, forced by the issue's no-behavior-change constraint. The shared `--authored-after`/`-before`/`--include-nulls` argparse trio lives in `_common.add_date_filter_args`. No schema change.
