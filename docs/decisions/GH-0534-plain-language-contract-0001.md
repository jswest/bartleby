# Findings carry a plain-language contract — prompt-level only, by design (issue #534)

> Source: [#534](https://github.com/jswest/bartleby/issues/534)

A finding's `body` is free-form markdown, and until #534 SKILL.md said nothing about how it should *read* — every Output-section contract was about provenance (cite the right chunk, `[^N]` syntax, verify support), none about legibility. Since the corpora are insider documents (rate cases, ethics agreements, disclosures), the default failure mode is register-mirroring: the agent absorbs the source's jargon into its prose, and the verbatim-delivery rule then guarantees that jargon is exactly what the human reads and what every future agent inherits. The fix is a `## Plain language` section in `bartleby/skill/SKILL.md`: write for an educated non-specialist (a curious outsider follows the whole finding without a glossary), gloss terms of art on first use rather than banning them, spell out acronyms, prefer plain syntax — and the rule is explicitly subordinate to the citation/honesty contracts (plain language is *how* you write, never *what* you claim).

The deliberate call is **prompt-level enforcement only**, a scoped exception to the repo's "enforce at script level, not prompt level" instinct (the memory-off rule). That instinct applies where a violation is mechanically detectable — a session id is or isn't excluded. "Plain language" is a judgment over prose; the candidate script-level gates were all rejected: a readability lint (Flesch / sentence-length / jargon word-list) is brittle on legitimately technical findings and false-positives on glossed terms; an LLM judge or rewrite pass adds a provider call to every `save_finding`; a hard readability gate would reject valid findings on a heuristic. The accepted cost is that the contract relies on the agent *recognizing* jargon (the stated heuristic: if a general-news reader wouldn't know the word, gloss it) — named in the issue as judgment, not a gate.

Scope rides the existing verbatim-delivery contract: because the body you save and the body you deliver are the same bytes, one rule on the body governs both the durable record and the chat reply — no separate delivered-answer rule needed. No validation changes in `save_finding`/`edit_finding`, no new columns, no `SCHEMA_VERSION` bump; docs-only.
