# `list_findings` / `read_finding` — finding read paths (issue #40)

> Source: [#40](https://github.com/jswest/bartleby/issues/40)

two skill scripts that browse and read findings, mirroring the `list_documents` / `read_document` pair but for the agent's own memory. The data was already intact (`findings.body` holds the whole markdown; `finding_citations` maps each finding to its cited chunks) — the gap was read access: `list_documents`/`read_document` are documents-only, and `search --findings` returns ranked *fragments*, not "show me finding N" or "what findings exist." `list_findings` enumerates newest-first (`ORDER BY finding_id DESC`) with `citation_count` and the `list_documents` pagination/`hint` idiom; `read_finding` returns the full body in the same shape `save_finding`/`edit_finding` emit (so the verbatim-echo contract holds), reusing `resolve_citations`. **Memory-off behavior diverges from `search` deliberately**: `search` silently drops findings from a mixed result list (the load-bearing invariant), but these finding-only commands raise an explicit `{"code": "MEMORY_OFF"}` error via a new `require_memory_enabled` guard in `_common.py` — a direct-intent "read my findings" failing silently would be more confusing than honest, and this deliberately reveals that findings exist (accepted tradeoff). No schema change; both wired into the dispatcher `SCRIPTS` tuple. Out of scope: filtering `list_findings` by session/tag, and any change to `search --findings`.
