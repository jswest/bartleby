# save_date backfills authored_date — and writes the summary row, not documents (issue #369)

> Source: [#369](https://github.com/jswest/bartleby/issues/369)

A new mutating skill script, `save_date --document <id> --date YYYY-MM-DD` (or `--clear`), lets an agent backfill or correct a document's `authored_date` when the date is in the document text but absent from metadata. It is a curation write in the `save_summary` / `assign_tag` mold: use it only when you have *read the evidence* for the date (a dateline, a "signed on" line, an export header), cite-don't-guess. The script validates the format, returns `{document_id, old_authored_date, new_authored_date}`, and rides the existing runner mutation seam (`run(mutates=True)` → one `with conn:`, plus the `log_call` audit row that records the attempt even on rollback) — no new transaction or audit machinery. No schema change; `SCHEMA_VERSION` untouched.

**It writes `summaries.authored_date`, not `documents.authored_date` — because that is the column every read path actually reads.** The issue text asserts "`documents.authored_date` already exists." It does not. The only `authored_date` column in the schema lives on `summaries` (added by `_upgrade_v…` as `ALTER TABLE summaries ADD COLUMN authored_date TEXT`); `documents` has no such column. Every agent-facing read path reads `s.authored_date` off the summary: `list_documents` (its `--sort date` and the `authored_date` field it reports), `scan`/`search` (the per-match `authored_date` that landed in #368, sourced from the `summaries` LEFT JOIN), and the `--authored-after`/`--authored-before` date filters in `_tags.py` (`s.authored_date >= ?` / `<= ?`, with the `IS NULL` undated-handling). Writing to a phantom `documents.authored_date` would set a value no read path could ever see — the document would stay undated through every filter, defeating the issue's own done-criterion ("the document immediately participates in `--authored-after/--before`"). The integration constraint from the omnibus (the write must be visible through #368's read paths) and the issue's stated column therefore diverge on a false premise; reconciling in favor of the column that is *read* is the only choice that satisfies the actual objective, so no park was warranted.

**A document with no summary is rejected (`NO_SUMMARY`), not auto-summarized.** `authored_date` is physically a summary-row column, so a document without a summary has nowhere to carry a date. The undated documents the issue targets are present in `list_documents` — i.e. they already *have* summaries with a NULL `authored_date` — so the repair path is a plain `UPDATE`. Rather than silently minting a placeholder summary (which would pollute `list_documents` titles/descriptions and overstep "smallest fix that fits"), the script tells the agent to run `save_summary` first. A non-existent document id is rejected with `DOCUMENT_NOT_FOUND`.

**Malformed dates are rejected loudly, unlike summarizer storage.** `normalize_authored_date` returns `None` on junk — correct for the summarizer (store NULL rather than something structured-but-wrong), but a footgun for a manual curation write, where a silent NULL would look like a successful set while quietly clearing the field. So `save_date` reuses `normalize_authored_date` for parsing but raises `INVALID_DATE` (non-zero exit, stdout `{"error","code"}` envelope) when it returns `None` for a `--date` value — the same loud-reject posture `validate_date_bound` already takes for the filter flags. `--clear` is the only sanctioned path to NULL, and `--date`/`--clear` are a required mutually-exclusive group.

Touches: new `bartleby/skill_scripts/save_date.py`, registration in `bartleby/commands/skill.py`, a scripts-table row + invoke example in `bartleby/skill/SKILL.md`, and `tests/test_skill_save_date.py` (valid write returns old+new; visibility through `list_documents`; malformed and impossible-calendar dates rejected; `--clear` nulls; `NO_SUMMARY`/`DOCUMENT_NOT_FOUND`; `--date`/`--clear` mutual exclusion).
