# Per-chunk view + quick-links from every chunk reference (issue #156)

> Source: [#156](https://github.com/jswest/bartleby/issues/156)

a chunk id was inert text (`chunk 1234`) everywhere it appeared — no way to *look at* a chunk and its surrounding context. Added a `/chunks/[id]` route plus an open-in-context affordance beside every chunk reference; `web/`-only, read-only, **no schema change**. **The view**: `getChunk(chunkId)` (`queries.js`) returns the target chunk plus its **±2 neighbors by `chunk_index` within the same `(source_kind, source_id)`** (one `BETWEEN` query off `idx_chunks_source`, target flagged via `is_target`), clamped at source boundaries; `+page.svelte` renders the target at full `--color-text` contrast on a `--color-token-dark` rail with the neighbors muted `--color-off` as context, each non-target label linking to its own `/chunks/<id>` so you can walk the stream. **Generic over source_kind** (the answer to the issue's open question): the neighbor window is within-source regardless of kind, and the back-link branches — document/summary/image chunks resolve to the document detail page via the existing `resolveSource` (same mapping citations/search use), a finding chunk links to `/findings/<source_id>`; unknown id → `error(404)` like the document/finding routes. **The quick-links**: a shared inline-SVG `CHUNK_ICON` (`lib/icons.js`) rides beside each chunk reference → `/chunks/<id>`. On search/scan cards (`ResultCard.svelte`) it's a plain `<a>` (native SPA nav). On finding citation chips (`findings/[id]/+page.svelte`) it's a **sibling `<button>`, not an `<a>`** — the chips are raw HTML inside `{@html}` and the layout's `marked` postprocess hook rewrites *every* `<a>` to `target=_blank`, so an anchor there would break in-tab routing; instead `renderChip` emits a `<span class="cite">` wrapping the chip + a `.cite-jump` button (DOMPurify keeps the `data-chunk-id` button, same as the cite-chip), and the existing click-delegation gained a `.cite-jump` branch that calls `goto()`. The chip's own click still opens the side iframe — the jump is a *separate* affordance. **Two small reuse moves** preempted by the simplify pass: the markdown-vs-literal snippet rule was hoisted out of `ResultCard` into `isMarkdownChunk` (`format.js`), shared with the chunk view; `renderChip`'s hand-rolled `.pdf`-stripper became the shared `stripExt` (so a `.docx`/`.html` chip strips its extension too). Verified end-to-end with Playwright against a live corpus — document/finding/summary chunks, boundary clamping, 404, both quick-link sites, and the preserved chip-iframe behavior — plus a clean `npm run build`.
