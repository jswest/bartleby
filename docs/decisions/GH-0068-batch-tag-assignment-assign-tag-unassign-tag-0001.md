# Batch tag assignment — `assign_tag` / `unassign_tag --documents <id,id,...>` (issue #68)

> Source: [#68](https://github.com/jswest/bartleby/issues/68)

every `bartleby skill <name>` call is a cold `uv run` + Python boot, so a per-document operation over a corpus multiplies into N process spawns. An agent doing one tagging pass made 63 `assign_tag` calls — slow enough that the cost actively shaped its approach. Fix: both scripts take a single tag and a comma-separated **list** of document ids (`comma_int_list` from `_common.py`, the same parser `read_chunks --chunks` / `scan --in-documents` use), amortizing interpreter startup over the whole set in one boot. The singular `--document` was **replaced**, not supplemented (no-backwards-compat rule) — a one-document call is `--documents 9`; the `tag` classifier keeps its own singular `--document`/`--all` scope since it's a different operation. Per-document resolution is one `WHERE document_id IN (...)` via the new `_tags.resolve_documents()` (which also dedups and preserves input order); ids with no document land in a `not_found` list and are skipped **without aborting** the rest of the batch — the per-document analogue of `tag`'s `failed` list. The flat single-pair output (`{document_id, file_name, tag_id, tag, assigned: true}`) became `{tag_id, tag, assigned: [{document_id, file_name}, ...], not_found: [...]}` (and `unassigned` for the inverse). Idempotency is unchanged (`assign()` is still `INSERT OR IGNORE`; `unassign()` still no-ops on an absent pair). No DB or schema change. Still not built: the optional generic batch/stdin "list of arbitrary operations in one boot" from the issue — the multi-document form removes essentially all the friction the issue cites, and a general op-stream is a much larger surface for speculative benefit.
