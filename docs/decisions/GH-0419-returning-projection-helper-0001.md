# `--returning <field>,...` projection across the four read scripts (issue #419)

> Source: [#419](https://github.com/jswest/bartleby/issues/419)

`search`, `scan`, `read_chunks`, and `list_documents` gained a `--returning chunk_id,document_id,...` flag that projects each row to exactly the named top-level fields, in the order given, through **one** shared helper ‚Äî `project_row(full, requested, whitelist)` in `_common.py` (plus `add_returning_arg` and the `comma_field_list` argparse type). The helper is the keystone other v0.9.4 sub-issues build on, so it's deliberately tiny and dumb: the caller builds the full whitelisted row dict, then `project_row` either returns `None` (no `--returning` passed ‚Üí caller keeps its own default/brief/verbose shape untouched ‚Äî the flag is **purely additive**, no envelope change, default output byte-for-byte unchanged) or returns `{field: full[field]}` in requested order. An unknown field raises `UNKNOWN_RETURNING_FIELD` with the valid set in `valid_fields` ‚Äî the JSON error envelope the skill contract demands, *not* argparse's exit-2 dump (the `comma_field_list` type can't see the per-script whitelist, so it validates only emptiness and defers the whitelist check to `project_row` at work() time, where a `SkillError` becomes the envelope).

**The whitelist is per-script, deliberately not global.** The four scripts return structurally different rows (a search hit carries `rank`/`score`/`normalized_score`/image locators; a `list_documents` row carries `title`/`has_summary`/`tag_values`; a `scan` match carries `text_length`/`section_heading`), so one shared field set would either be a meaningless union or force every script to carry null columns it has no concept of. Each script owns a module-level `*_FIELDS` constant; the helper takes it as a parameter. This mirrors the `--brief` precedent (GH-0105: "one flag name, per-command semantics ‚Äî the field sets have nothing in common") ‚Äî but unlike `--brief`, the *mechanism* (validate-and-select) genuinely is shared, so here the single helper earns its keep where a shared `--brief` projection didn't.

**`chunk_id` and `document_id` are selectable in every projection ‚Äî the load-bearing point.** Both lead every whitelist so an agent can recover a citable id in one call instead of a re-read round-trip. Two non-obvious calls:

- **`scan --count-by document` exposes a `chunk_id`** even though a per-document histogram row has no natural single chunk. It's `MIN(c.chunk_id)` over that document's matching chunks ‚Äî a *representative* citable handle, folded into the existing `GROUP BY` so it costs no extra query. It is selectable via `--returning` but **not** in the default histogram row (which stays `{document_id, file_name, chunk_count}`): the histogram contract is about counts, the chunk_id is the escape hatch when you want to cite. `--count-by '/regex/'` instead **rejects** `--returning` (`RETURNING_NOT_APPLICABLE`): a captured value spans documents, so its `{value, count}` buckets carry no citable id at all ‚Äî surfacing a fake one would be dishonest.
- **`document_id` is honest-null off non-document chunks.** In `search` / `read_chunks --chunks` / `read_chunks --around-chunk`, `document_id` = `source_id` only when `source_kind == "document"`; summaries span one document but aren't it, images can span many, findings have none ‚Äî so it's `null` there, never faked (same posture as `authored_date`). In `list_documents`, where rows *are* documents, `document_id` is an alias of the default row's `id` (same value; `id` stays selectable too so the default field name still works). In `read_chunks --document` it's the constant document id of the page.

**`--returning` overrides `--brief`** (and `list_documents`' verbose tier): when both are passed, the explicit field selector wins. `read_chunks --document`/`--around-chunk` default rows stay locator-light (no `source_*`/`file_name` ‚Äî the envelope names the document/target once), but `--returning` can pull the full set in any mode since every row carries it underneath. No schema change; additive-only; per-script whitelists documented in each script's `--help` `Output:` block and summarized in `SKILL.md`.
