# Per-unit ingest provenance + config-drift warnings (issue #171)

> Source: [#171](https://github.com/jswest/bartleby/issues/171)

once #164 made ingest resumable across separate `scribe` invocations, a single corpus could be parsed / captioned / summarized by *different runs under different config* (half docling / half pdfplumber, captions from two VLMs) with **no record of which run produced which unit** — corrupting the multi-model eval workflow, and captions in particular recorded *no* model at all. This captures provenance per unit and warns on drift. (1) **`ingests` table — one row per invocation**: `run_id`, `started_at`/`finished_at`, `config_json`, `bartleby_version`, `schema_version`. `Writer.begin_run` inserts it at the top of a run (storing `self.run_id`), `finish_run` stamps `finished_at` in the `finally`. (2) **Per-unit stamping (Option B — each unit carries its producing run)**: nullable `ingest_run_id` FK columns on `documents` (parse run), `summaries` (summary run), and **`chunks`** — the chunk column covers *both* parse chunks and image-caption chunks in one column, since captions are `source_kind='image'` chunks, so it answers "which VLM captioned this figure" / "recaption everything from model X", the real gap. The `Writer.persist_*` methods thread `self.run_id` through the INSERTs and the typed `db.chunks` helpers (`insert_*_chunks` gained a nullable `ingest_run_id`; `insert_finding_chunks` deliberately did *not* — findings come from a research session, not an ingest run, so the column stays NULL there, as it does on any Writer that never opened a run). (3) **Config snapshot is secret-stripped at one chokepoint**: `config.redact_config` drops every key whose name contains `api_key`/`token`/`secret`/`password`/`credential` (match-by-name so a *newly-added* credential field is stripped by default, not leaked until denylisted) — **API keys must never land in the project DB**. The snapshot is the file config overlaid with the run's CLI-resolved converters/provider/model, serialized sorted-key so a later compare is a plain string diff. (4) **Drift is warn-only**: `config.config_drift` diffs the new snapshot against the most recent prior `ingests` row (read *before* this run's row is inserted, so a run never compares to itself) field-by-field — *every* field, not a curated subset — and `scribe` emits one `console.warn` per differing field. You knowingly continue or stop; it never blocks (a mixed-config corpus is sometimes intentional). Units ingested before this feature keep NULL provenance — it's forward-looking. **Schema handling — held at v8 for the #169 omnibus**: the `ingests` table and the three `ingest_run_id` columns are additive DDL, but `SCHEMA_VERSION` is *not* bumped here. The whole concurrent-ingestion omnibus (#169) releases as a unit, so it takes a single v8→v9 bump with one consolidated `_upgrade_v8_to_v9` (CREATE `ingests` + three `ALTER … ADD COLUMN` — SQLite permits the inline FK on ADD COLUMN because the default is NULL; columns are appended at end of each table so the fresh DDL matches the ALTER output) at release, rather than each sub-issue bumping in turn (#171→v9, #165→v10, …). Consequence during dev: a fresh DB carries these structures at v8, but an *existing* v8 corpus must re-ingest to gain them (no upgrade entry exists yet, so `project upgrade` is a no-op for them). **Deliberately NOT built**: the version bump + upgrade-chain entry (deferred to the #169 release), a per-run override of `max_workers`-style knobs (no new behavior here, just provenance), blocking-on-drift (warn-only by design), backfilling provenance onto pre-feature units (forward-looking), and surfacing `ingests` in `project info` or the web UI (the table is queryable; surfacing is a separate concern).
