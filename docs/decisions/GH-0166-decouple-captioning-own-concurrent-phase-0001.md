# Decouple captioning into its own concurrent phase (issue #166)

> Source: [#166](https://github.com/jswest/bartleby/issues/166)

#164 moved the VLM call *out* of parse (parse records uncaptioned image rows; a later stage captions them), but that stage still captioned **serially on the main thread** — dozens of VLM calls per document, one at a time, the #162 benchmark's 29%-of-wall-clock stage run sequentially. This makes captioning concurrent. Landing alongside #167 (which lifted summarization into its own post-drain pass), `scribe.main` is now **three phases behind the single Writer: parse-all (#165 pool) → caption-all (this issue) → summarize pass (#167)**; `_DocUnit` carries each parsed/resumed doc from parse into the caption phase. The phased shape (rather than an overlapped parse↔caption pipeline) is deliberate — consistent with #167's pass, and avoids entangling #170's progress rework; parse is only ~12% of wall-clock, so the lost overlap is a sliver next to making the 29% caption stage concurrent at all. **(1) The thread/main-thread split is load-bearing.** `_caption_image` split into `_analyze_image` (load JPEG + Tesseract + VLM — pure analysis, no DB, no embed; the GIL-releasing network/IO work) and `_caption_from_analysis` (embed the caption chunk + package for the Writer). `_caption_all` runs `_analyze_image` across a `ThreadPoolExecutor`, but **embedding and every DB write stay on the main thread** — the one per-process `SentenceTransformer` isn't safe to call from several caption threads at once, and the apsw Writer is single-owner. So only OCR+VLM fan out; the embed+persist tail is serialized, preserving #165's "all concurrency lives upstream of the DB" invariant. **(2) Dedup by image row.** `_caption_all` collects the unique uncaptioned rows across all units (keyed by `image_id`, so an image shared by several documents is analyzed once) and skips capped rows. Caption incompleteness is recomputed from the DB after the phase (`bool(writer.uncaptioned_images(document_id))` per unit), which correctly counts a shared failed image against *every* document holding it. Idempotency/resume unchanged (only `analysis_json IS NULL` rows are touched; `persist_caption`'s guard makes a re-apply a no-op). **(3) One pipeline, one swappable scheduler** — mirrors `pool.py`'s `max_workers <= 1` seam: at `caption_workers <= 1` the work runs inline (the path `--timings` forces, for clean per-document caption seconds), above it across the thread pool; both call the same `_analyze_image`. **(4) `caption_workers` is config-only** (default `DEFAULT_CAPTION_WORKERS=4`; new `bartleby config` prompt in the vision block, no CLI flag), a flat count rather than parse's RAM-derived auto-pick because captioning is network/IO-bound — 4 is safe for both a single-GPU local Ollama (which serializes anyway) and a rate-limited cloud provider. **(5) Timing fidelity preserved.** `--timings` pins `caption_workers=1`; parse + caption seconds accumulate onto each `_DocUnit.stages` (only when timing — zero overhead otherwise), seed #167's `doc_timings` (keyed by `document_id`) after the caption phase, and the summarize pass adds its stage there — so the per-doc record stays one-line-per-doc and #162's benchmark / `docs/BENCHMARKS.md` stay comparable. **(6) Progress UX kept minimal** — one bar relabeled per phase (Parsing → Captioning images); the summarize pass adds its own bar (#167), and the richer per-worker "run of show" stays #170's. **Deliberately NOT built**: an overlapped parse↔caption pipeline (see above); the provider Batch API for captions (#168). No schema change.
