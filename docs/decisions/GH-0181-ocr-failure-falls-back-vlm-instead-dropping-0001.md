# OCR failure falls back to the VLM instead of dropping the image (issue #181)

> Source: [#181](https://github.com/jswest/bartleby/issues/181)

the image captioner runs Tesseract first as a cheap text-image *classifier*, then the VLM as the actual captioner. But a Tesseract *crash* (an unwritable/locked-down `TMPDIR` — the legible `RuntimeError` #43 added still *raises* — or any unexpected `pytesseract` error) propagated out of `analyze` and aborted the whole image, so the VLM (which never needed OCR) never ran and the image got **no caption at all**. Same trap as #80/#164: a non-essential stage failing quietly and reading as "done" — on the #162 benchmark it made captioning look nearly free because it had silently dropped every image. Fix is one place: in `bartleby/ingest/images.py`'s `analyze`, the *fresh* `ocr_module.run` is wrapped so any exception degrades to "couldn't classify → not a text image" (`ocr_result` stays `None`, the `_is_text_image` guard gains a `None` check) and falls through to the VLM. A `prefetched_ocr` passed down from a sparse-page render is *not* wrapped — the caller already computed it successfully; only the fresh run can throw here. `ocr_module.run` itself keeps raising — the **caller** decides to degrade, so the page-level OCR path is unaffected. A broken Tesseract warns **once per process** (a module-level flag, via `console.warn`) rather than once per image, so a 9k-doc corpus surfaces the misconfiguration without spamming it. Net: a busted Tesseract just routes every image straight to the VLM — slower (the text-image shortcut is lost) but lossless. No schema change.
