# Disable load_extension() after _attach (issue #330)

> Source: [#330](https://github.com/jswest/bartleby/issues/330)

`_attach` in `bartleby/db/connection.py` (apsw, not stdlib sqlite3) called `conn.enable_load_extension(True)` then `sqlite_vec.load(conn)` but never turned extension loading back off, leaving the SQL `load_extension()` function callable for the connection's whole lifetime on both call sites (`open_db`, `init_db`). All query text is parameterized today, so this was defense-in-depth, not a live exploit — but the standard idiom is enable → load → **disable**, and the fix is one line: `conn.enable_load_extension(False)` immediately after `sqlite_vec.load(conn)`. Disabling further *loading* does not unload the already-loaded sqlite_vec extension — confirmed by the suite (vec-dependent tests stay green) and by the new focused test, which asserts `SELECT load_extension('x')` now raises `apsw.SQLError` ("not authorized") while `vec_version()` still returns. We deliberately chose the minimal one-liner over a `try/finally` wrapper: `_attach` does no work between load and disable that can fail in a way that would leave loading enabled, so a wrapper would add ceremony without protection. Out of scope: any broader SQLite hardening beyond this single disable call. No schema change.
