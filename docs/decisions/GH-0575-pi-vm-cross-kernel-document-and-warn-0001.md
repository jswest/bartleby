# A containerized (pi-vm) session needs exclusive host access to its corpus: document-and-warn

The cross-kernel WAL clash (a host `bartleby serve` and a pi-vm agent touching one corpus at once) is addressed by **documenting the constraint and warning at the boundary**, not by changing cross-kernel behavior. The breakage is **strictly cross-kernel, not concurrency**: SQLite's WAL wal-index (`bartleby.db-shm`) is only coherent within one kernel, so the Linux guest and the macOS host sharing one `/corpus`-mounted WAL DB is the unsupported case. Same-kernel concurrency is supported and tested — `busy_timeout=5000` set before `journal_mode=WAL` (`connection.py:62-64`), the mutating seam on `BEGIN IMMEDIATE` (`skill_runner.py:218`), embedding hoisted outside the transaction (#340), and `tests/test_skill_concurrent_writes.py` landing six concurrent writers with zero BusyError (#562/#339) — so multiple host sessions, or host `serve` + a host session, are fine. The bug has two faces: the loud `database disk image is malformed` 500, and the quiet stale-reads-until-restart (a non-erroring host `serve` silently hides new findings the container writes). Both clear once the container is idle and its writes checkpoint.

The fix: a strengthened runbook section (`docs/pi-vm-runbook.md`) stating the plain operational rule — while a pi-vm session runs against a corpus, do not access that corpus from the host; stop `bartleby serve`, don't start a host session, view between runs — and naming both faces; a concise stderr warning at pi-vm launch (`scripts/pi-vm/entrypoint.sh`); and one brief `console.warn` line at `bartleby serve` startup pointing at the runbook (kept minimal because host-only serve is the common, fine case and must not be drowned in noise).

Rejected, with the reason each is the wrong fix: **(a) in-container serve** — there is normally no container, so this reuses the wrong surface for the common case; **(b) checkpoint+reopen on the reader** — papers over only the staleness face and leaves the malformed face; **(c) a non-WAL journal mode** — taxes *every* corpus (losing WAL's concurrent-reader/writer story) and leans on cross-mount POSIX locks. Supersedes #558. No schema change, no journal-mode change, no in-container serve.
