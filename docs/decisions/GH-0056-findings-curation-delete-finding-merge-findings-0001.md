# Findings curation — `delete_finding` + `merge_findings` (issue #56)

> Source: [#56](https://github.com/jswest/bartleby/issues/56)

findings previously had a write path (`save_finding`/`edit_finding`) and read paths (`list_findings`/`read_finding`) but no *curation* path, so duplicates and zero-citation drafts accreted with no way out — the asymmetry the issue called out against tags' full lifecycle (`delete_tag`/`merge_tags`). Two new skill scripts close it. `delete_finding --finding <id>` retracts one finding: `delete_chunks_for(conn, "finding", id)` clears its body chunks (+ `chunks_fts`/`chunks_vec`) and the `findings` row delete cascades `finding_citations` via the FK. `merge_findings --from <ids> --into <id> --body-file <path>` collapses a cluster: the `--into` target is kept (its `finding_id`/provenance/authoring session survive), the agent **authors** the consolidated body (validated by the same `[^N]` rules as `save_finding`), the target's body+citations are rebuilt, and the `--from` sources are deleted. Both touch only finding rows — cited *document* chunks are never affected, because findings are derivative hints, not evidence. Mutations run inside `with conn:` for atomicity (apsw nests the helpers' own transactions via savepoints). No schema change (`SCHEMA_VERSION` stays 6). **Deliberately NOT built** (scoped out by the user, against the issue's broader menu): the dedup-on-save similarity check on `save_finding` (no `--force`), and the supersede relationship (`superseded_by` column / `list_findings --include-superseded`) — the issue floated both, but delete + author-merge cover the motivating 5/6/7 and 20–23 cluster cases without a schema change or a new soft-delete state to reason about. Mechanical concat-merge (append source bodies) was rejected in favor of the authored body: auto-concatenating four report iterations yields an ugly low-quality finding, and the agent rewriting one clean consolidation is the actual goal. Neither command gates on `memory_enabled` — they act on an explicit `finding_id` like the `save`/`edit` write siblings (only the *reveal* commands `list_findings`/`read_finding` raise `MEMORY_OFF`). The shared "read body file → validate citations" logic (duplicated across `save_finding`/`edit_finding`, and a third copy waiting in merge) was hoisted to `_common.load_finding_body` / `_common.validated_replacement` in the same change.
