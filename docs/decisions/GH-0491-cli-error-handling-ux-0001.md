# CLI error-handling & UX consistency: expected error → exit 1 on stderr (issue #491)

> Source: [#491](https://github.com/jswest/bartleby/issues/491)

The CLI's error surface had drifted: some expected user errors exited cleanly while siblings leaked tracebacks, error text landed on **stdout** (breaking `2>/dev/null`), an input validator silently inverted its own meaning, and the config wizard carried dead code. This bundle enforces one convention end-to-end — **every expected user error exits 1 with a one-line message on stderr, never a traceback; success/tabular output stays on stdout; wizard prompts validate their own bounds.** All additive, no `SCHEMA_VERSION` bump.

The eight fixes, by theme:

**Expected errors → clean exit.** `bartleby scribe`'s expected failure modes — no active project (`RuntimeError`), an invalid configured converter or typo'd `--only` (`ValueError`), a missing path (`FileNotFoundError`) — propagated as uncaught stack traces while create/info/delete caught theirs. The `cli._scribe` dispatcher now wraps the `scribe_main` call in `try/except (ValueError, RuntimeError, FileNotFoundError)` → `console.error` + `exit(1)`. Wrapping at the dispatcher (rather than threading the catch through scribe's own `main`) keeps the pipeline body raise-on-error and centralizes the one translation point. `project upgrade` called `validate_project_name` bare — an invalid name traceback'd where create/info wrap it; it now mirrors create's `try/except ValueError`.

**stderr / convention.** `project` / `session` / `logs` printed errors through a module-level stdout `Console` before `exit(1)`. All error-exit paths now route through the existing `bartleby.lib.console.error` helper (the shared stderr Rich console already used by `ready.py`/`serve.py`/`scribe.py`), so `2>/dev/null` suppresses them. The module-level `_console` is retained where it renders **success tables** (`project list`/`info`/`verify`, the `logs` table, session lines) — those stay on stdout by design.

**Input validation.** `bartleby logs --limit` feeds `LIMIT ?`; SQLite reads a *negative* LIMIT as unbounded, so `logs --limit -5` silently dumped every row. `logs.main` now rejects `limit < 1` up front. In the wizard, `ocr_min_confidence` advertised "(0-100)" but was collected via `_prompt_positive_int` (`n > 0`), rejecting the meaningful `0` ("always trust OCR") while accepting `250`. A small `_prompt_bounded_int(min_value, max_value)` helper now bounds it to 0–100. (`_prompt_positive_int` was deliberately left untouched — folding the two would have rippled through its other callers, beyond this bundle's smallest-fix scope.)

**Silent-discard / dead code.** `_resolve_llm_provider` returns early when `summary_depth != "one-shot"`, *before* reading `--provider`/`--model`, silently discarding either override; it now `console.warn`s in that branch when an override was passed. The wizard's `main()` rebuilds config from an empty dict, so every `config.pop(k, None)` over never-added keys was dead — all removed (keeping the one live `config["summary_depth"] = "none"`); the `if ollama: pop else: prompt` shapes collapse to `if not ollama: prompt`. Finally, `_prompt_api_key`'s default help advertised `WSJPT_API_KEY`, but `ensure_provider_env`/the provider read only `GEMINI_API_KEY`; both wsjpt-capable call sites now pass explicit `help_text` from a new `_api_key_help(provider)` that names the correct env-var.

A new `tests/test_cli_errors.py` pins the behavioral half: scribe's three expected exceptions become exit-1-on-stderr (asserted via the repo's `monkeypatch console.error` idiom, since the shared Rich console binds its stream at import time), `project upgrade` rejects a bad name, `logs --limit` rejects -5 and 0, the unknown-session error routes to stderr, the override-ignored warning fires (and doesn't when no override is passed), `_prompt_bounded_int` accepts 0 and rejects 101/250/-1, and the wsjpt help names `GEMINI_API_KEY`. One pre-existing test (`test_upgrade_refuses_below_chain_first_step_without_mutation`) that captured the refusal message off the stdout `_console` was updated to capture `console.error` instead — a direct consequence of the stderr routing. Suite green at 1032 passing.
