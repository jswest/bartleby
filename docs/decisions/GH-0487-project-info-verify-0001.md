# `project info --verify` is a read-only corpus integrity audit over one shared check module (issue #487)

> Source: [#487](https://github.com/jswest/bartleby/issues/487)

`bartleby project info` reported counts but nothing audited whether the DB was *structurally* consistent — drift between `chunks` and its FTS/vec mirrors, orphaned `chunks.source_id`, missing tables, or a stamped schema version the on-disk shape didn't match would surface only as a raw apsw traceback mid-query, if at all. `--verify` adds a read-only audit that exits 0 on all-pass / non-zero on any failure, printing a per-check `PASS`/`FAIL` line; the existing counts table still prints first. Four checks: **tri_table_sync** (`chunks` / `chunks_fts` / `chunks_vec` agree), **no_orphan_chunks** (every `source_id` resolves to a live parent per `source_kind`), **schema_matches_stamp** (the tables/indexes the stamped `schema_version` implies exist), **failed_ingests_sanity** (a `parse`-stage failure can't coexist with live document chunks for the same file). The audit always runs the full battery — one bad check doesn't short-circuit the rest — so a single invocation names every problem.

The tri-table check is the load-bearing one and it already existed, as `tests/_skill_fixtures.py:assert_chunk_tables_consistent`: FTS5 external-content drift is invisible to any non-MATCH read (those are satisfied *through* `chunks`), so the only catch is FTS5's own `'integrity-check'` command in the two-argument `rank=1` form — the one-argument form is a no-op for content/index drift in this SQLite. Rather than reimplement that subtlety in the command and let two copies drift, the logic was lifted into a new shippable `bartleby/integrity.py`; the test fixture now delegates to `check_tri_table_sync` and keeps only a thin `assert result.passed`. One implementation, consumed by both surfaces — exactly the de-duplication the issue asked for, and no new abstraction beyond that single module.

Two deliberate calls. **The verify path opens its own raw connection** (vec attached, no version gate) instead of going through `open_db`/`get_project_info`, because `open_db` *raises* on a schema-version mismatch — which is precisely the corruption `schema_matches_stamp` is meant to *report*. Gating the audit behind the gate it audits would make the most important failure unobservable; so under `--verify` a `RuntimeError` from `get_project_info` is swallowed (the counts table is skipped) and the checks run anyway and name the mismatch. **The `schema_matches_stamp` check compares the DDL-declared objects as a subset** of `sqlite_master`, not an exact set, because the FTS5 and vec0 virtual tables spawn shadow tables not named in the canonical DDL — requiring presence of the declared objects catches missing-table corruption without false-positiving on the shadows.

Everything is read-only: no transaction is opened, no corpus row mutated. The FTS5 `'integrity-check'` is a vacuous write that persists nothing. That last point forced one guard adjustment: the polymorphic-chunks chokepoint test (`test_chunks_chokepoint.py`) statically forbids `INSERT INTO chunks_fts` outside `bartleby/db/chunks.py`, and the integrity-check command is syntactically such an insert. The guard's regex now exempts only the FTS5 *command* form — where the first column named is the table itself (`chunks_fts(chunks_fts, …)`), the command convention, never a real row write — while a genuine row insert (`chunks_fts(rowid, text, …)`) still trips it. The exemption is pinned by both a positive and a negative assertion in the guard's own tests, so it can't silently widen.
