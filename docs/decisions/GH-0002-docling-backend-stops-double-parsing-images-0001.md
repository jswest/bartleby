# Docling backend stops double-parsing for images (issue #2)

> Source: [#2](https://github.com/jswest/bartleby/issues/2)

the docling PDF path ran a *full second parse* — a `pdfplumber_pipeline.convert` over the same file — purely to enumerate embedded images for the vision pipeline, on top of docling's already-slow pass. Embedded pictures now come out of the **same docling pass**: `convert()` gained `extract_images`, and when set the converter runs with `generate_picture_images=True` at `images_scale = 150/72` (150 DPI, parity with pdfplumber's `PAGE_RENDER_DPI`); a new `_iter_picture_images(doc)` walks `DoclingDocument.pictures`, pulls each raster via docling's own `pic.get_image(doc)`, takes the page from `pic.prov[0].page_no`, and assigns a per-page 1-based `image_index_on_page` (the image-pipeline convention) — surfaced as `DoclingResult.images: list[DoclingImage]` (png_bytes + page + index), which `_ingest_pdf_docling` maps straight to `_ImageRoute`s. **Picture-image generation is gated on need**: `extract_images = vision_provider is not None`, and `_converter` is keyed on a `with_picture_images` bool (`lru_cache(maxsize=2)`) so text-only ingests pay nothing and the two converter variants coexist without cache thrash. The now-orphaned `sparse_text_threshold` param (it only fed the deleted side-pass) is dropped from `_ingest_pdf_docling`. **Scope — embedded rasters only**: pictures docling can't rasterize (vector figures, some scanned-page art) return `None` from `get_image` and are skipped; capturing those is the separately-tracked vector-figure follow-up, so the issue's sketched pypdfium2 bbox-render fallback was deliberately *not* built. The extracted image *set* can differ slightly from the old raw-XObject enumeration (docling returns layout-detected pictures) — expected. Real docling was exercised against a generated text+image PDF (`extract_images=True` → one 150-DPI picture with png bytes; `False` → zero, no wasted work); the pytest layer keeps the repo's fake-`SimpleNamespace` docling convention (no real models in CI) and the scribe test monkeypatches `pdfplumber_pipeline.convert` to raise, failing loudly if any second parse ever returns. No schema change.
