# Parallelize the summarize pass (issue #188)

> Source: [#188](https://github.com/jswest/bartleby/issues/188)

#167 lifted summarization into its own post-drain pass but left it **sequential on the main thread**, earmarking the throughput win for the provider Batch API (#168). With #168 deferred out of the v0.8.0 omnibus, this takes the cheap in-process win now and makes the ≈59%-of-wall-clock stage concurrent. `_summarize_all` fans the per-doc `summarize()` LLM call out across a `summarize_workers` `ThreadPoolExecutor`, mirroring caption's #166 split — the network-bound LLM call on worker threads, **every Writer call (`summary_input`, `persist_summary`, `record_failure`, `clear_failure`, `is_capped`) on the main thread**, preserving #165's "all concurrency lives upstream of the DB" invariant rather than softening it. **One deliberate divergence from caption.** A caption's payload is an on-disk JPEG the worker reloads itself; a summary's payload is assembled from the DB by `Writer.summary_input` — a Writer-owned read that can't move off the main thread. So instead of caption's submit-all-futures, the pool **bounds in-flight work to `summarize_workers`**: fetch each document's input on the main thread, submit, and top up one as each completes (`islice` prime + `next(it, None)` refill via `wait(FIRST_COMPLETED)`) — holding the old sequential pass's flat memory footprint instead of materializing every document's text at once (the omnibus targets a 9k-doc corpus). **`summarize_workers` is config-only** (default `DEFAULT_SUMMARIZE_WORKERS=4`; new `bartleby config` prompt in the summarization block, no CLI flag), a flat network/IO-bound count exactly like `caption_workers`; `--timings` pins it to 1 and takes the inline path for clean per-doc summarize seconds. The cap-skip guard (`is_capped` → surface as still-incomplete, never retry) moves to the work-list build; the zero-chunk / already-summarized exclusions already live in `documents_needing_summary` (#80), so the old per-doc `_summarize_if_missing` collapses into the pass (its zero-chunk unit test becomes a `documents_needing_summary` exclusion test). **Supersedes #167's "concurrent summarize pool — deliberately NOT built"**; #168 (Batch API) remains the heavier provider-side lever this pass is still the seam for. No schema change.
