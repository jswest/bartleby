# Web UI search reuses the skill scripts via subprocess (issue #65)

> Source: [#65](https://github.com/jswest/bartleby/issues/65)

the SvelteKit UI gained `/search` — ranked `search` (FTS+semantic RRF) and enumerating `scan`, with source-kind / `--tag` / `--in-documents` / `--add-context` filters and per-hit chunk expansion (`read_chunks`). The web does **not** reimplement any ranking: semantic search needs BGE query embeddings that only exist in Python (`search.py` shells out to `bartleby embed`; the "no daemon" call stands), so a JS port could only ever do FTS and would fork the logic. Instead a single server seam, `web/src/lib/server/skill.js:runSkill(name, args)`, spawns `bartleby skill <name> … --project <active>` with list-form argv (no shell), parses the JSON envelope, and surfaces `{error, code}` as a `SkillError`; routes render what the scripts already emit. Cheap browse queries still use the read-only `better-sqlite3` connection (`db.js`), so the DB is read-only *for browsing* but the search subprocesses open it read-write (session + audit rows). **Session model for a non-agent caller**: skill scripts always run inside a session, and findings visibility is gated on `memory_enabled`, but sessions were designed to track *agents*, not the human at a browser. So `runSkill` sets `BARTLEBY_SESSION_NAME=web-reader`; the runner (`skill_runner.py`) honors that env var by calling the new `session.ensure_named_session(project, name)` — a find-or-create of one durable, memory-enabled session by fixed name that **never reads or writes the `.active_session` pointer**. Net: the web always sees findings, never hijacks (or is hijacked by) the agent's active session, and its audit-log reads are attributable to a recognizable session. The env var (not a `--session` CLI flag) was chosen so the override stays off the agent-facing `--help` surface and needs no per-script parser change. Latency (BGE cold-load per query) is accepted as-is with a loading state in the UI. No schema change (`SCHEMA_VERSION` stays 6). Deliberately not built: writing from the web (save/edit findings, tagging), an embedding daemon, and deep file links for summary/image hits (the `search` output doesn't expose their `document_id`, and surfacing it would be a script change out of scope here).
