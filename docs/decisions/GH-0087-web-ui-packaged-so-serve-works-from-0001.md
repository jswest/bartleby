# Web UI packaged so `serve` works from an installed wheel (issue #87)

> Source: [#87](https://github.com/jswest/bartleby/issues/87)

`bartleby serve` was structurally run-from-checkout-only — it located the SvelteKit UI via `Path(__file__).resolve().parents[2] / "web"` (repo root), but `web/` was a *top-level* dir excluded from the build (`packages.find` only takes `bartleby*`), so an installed wheel resolved `WEB_SRC` to a non-existent `site-packages/web` and hard-failed with `FileNotFoundError`. Every other subcommand worked from an install, making this surprising. Fix: the UI moved from `web/` → **`bartleby/web/`** (now package data, declared via `[tool.setuptools.package-data]` scoped to `web/package.json`, `web/svelte.config.js`, `web/vite.config.js`, `web/src/**/*` — recursive `**` needs `setuptools>=62.3.0`, so the build req was bumped). `node_modules`/`.svelte-kit`/`build` are deliberately **not** bundled (dev/build artifacts regenerated at runtime), nor is `package-lock.json` (untracked; npm regenerates it on first install in `~/.bartleby/serve`). `serve.py` now resolves `WEB_SRC` via `importlib.resources.files("bartleby") / "web"` (identical from a checkout or a wheel). The HMR dev loop is **conditional, not removed**: a new `_is_source_checkout()` checks for a `pyproject.toml` two levels above `WEB_SRC` (repo root in a checkout/editable install; absent under `site-packages`), and `_sync_web(..., symlink_src=...)` *symlinks* `src/` for live vite HMR from a checkout but *copies* it from an installed wheel (the packaged source is read-only — symlinking it would be pointless). Verified by installing the wheel into a throwaway venv and confirming the previously-crashing path now syncs. Deliberately NOT changed: a packaged install still runs `vite dev` (not a `vite build`/preview server) — switching the production install to a built server is a separate concern the issue scoped out. No schema change.
