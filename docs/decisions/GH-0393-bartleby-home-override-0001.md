# `BARTLEBY_HOME` overrides the state-dir root; bartleby owns the mechanism, tooling owns the policy.

Bartleby's entire on-disk state — `projects/`, `config.yaml`, scratch, the served UI under `serve/` — roots at `~/.bartleby`, resolved through `bartleby_dir()` in `config.py` (`projects_dir()` / `config_path()` derive from it). That resolver honors a `BARTLEBY_HOME` env override: a generic state-dir knob in the spirit of `PGDATA` / `GIT_DIR` / `XDG_DATA_HOME`, earning its place for test isolation, CI, and running more than one corpus root — independent of any agent workflow. **The product stops there: it knows nothing about git worktrees, ultraship, or coding agents.** Resolution is **lazy on every call, never a module constant** — an eager import-time read freezes the path, so a `BARTLEBY_HOME` set *after* `bartleby` is imported (the natural in-process pattern) becomes a silent no-op and writes still hit the live namespace. That exact freeze is the GH-0393 bug: an unattended ultraship critic's ad-hoc verification script set `os.environ["BARTLEBY_HOME"]` and called `init_db("probe")`, but no override existed, so it wrote a (deliberately FTS-corrupted) corpus into the developer's *live* `~/.bartleby/projects/probe`. The **policy** — "an agent working in a worktree gets a sandbox" — lives in tooling, not product code: the `bartleby-home-sandbox.sh` `SessionStart` hook sets `BARTLEBY_HOME` to a per-worktree dir under `~/.bartleby-worktrees/<hash>` whenever the session's repo root is a *linked* worktree (`.git` is a file, not a directory), and leaves the primary checkout on live `~/.bartleby` — so it is scoped to agent sessions, not the user's own. ultraship's SKILL.md additionally instructs players never to drive the real CLI / `serve` / `project upgrade` / `ingest` against live corpora. Tests get the same isolation for free via one autouse `conftest` fixture pointing `BARTLEBY_HOME` at a per-test tmp dir — replacing ~9 copies of per-file `PROJECTS_DIR` monkeypatching and closing the "a new test file forgets to isolate" hole the old opt-in fixtures left open. The hook **degrades safe**: where the harness doesn't run `SessionStart` / expose `$CLAUDE_ENV_FILE`, isolation simply isn't automatic (behavior reverts to live `~/.bartleby`); it never misfires onto the wrong directory. The JS serve layer (`web/src/lib/server/db.js`) reads `BARTLEBY_HOME` once at launch rather than lazily — correct there, because `serve` is one long-lived process whose env is fixed when it starts (issue #393).
