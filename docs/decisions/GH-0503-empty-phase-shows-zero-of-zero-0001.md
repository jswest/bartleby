# An ingest phase with no work shows `0/0` (finished), not `—` (issue #503)

> Source: [#503](https://github.com/jswest/bartleby/issues/503)

The scribe run-of-show header renders each phase's tally as `done/total`, falling back to `—` for any phase whose denominator isn't yet known — its read is "this phase hasn't run." But the caption and summarize passes early-return *before* calling `phase.start()` whenever there's no work: an all-text ingest has no images to caption, and a fully-resumed (or already-summarized) run owes no summaries. On those very common runs the header sat at `caption —` / `summarize —` to the end, reading as "skipped" when in truth the phase ran and found nothing to do — a false negative every plain-text ingest hit.

The fix emits the denominator on those early-return branches so the phase reads as finished. `ScribeProgress._start` already maps a zero total to a *known*, already-complete phase (`done == 0 >= total == 0` → the `dim green` finished style with a `0/0` tally), so the branches just call `phase.start(0)` — `caption._caption_all`'s empty-pending return, and `commands/scribe.py`'s `if not pending` guard around the summarize pass, which short-circuits `_summarize_all` entirely. The vision-disabled caption branch is different: its rows exist but are declined, so it reveals the real denominator and advances it (`start(N)` then `advance(N)`) to finish at `N/N` rather than a stuck `0/N`. No schema change, no new API — the progress layer's existing `start(0)` semantics carry the whole fix; the only new code is the calls that were missing.
