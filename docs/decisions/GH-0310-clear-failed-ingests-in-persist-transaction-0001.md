# Clear `failed_ingests` inside the persist transaction, not after it (issue #310)

> Source: [#310](https://github.com/jswest/bartleby/issues/310)

The success path used to persist a unit (`persist_parse` / `persist_caption` / `persist_summary`) and clear its `failed_ingests` row (`writer.clear_failure(...)` at the `ingest/parse.py` / `caption.py` / `summary.py` call site) as **two separate transactions**. A crash between them on a previously-failed unit left the unit durably complete but its failure row alive ŌĆö and because resume work-lists derive from *completeness* (a captioned image / summarized doc is never re-offered), the unit never re-ran, so nothing ever cleared the row: a finished unit read as failed forever in `_report_failures` and `project info`. The documented invariant ("a skipped unit must never read as a completed one") violated in reverse.

**Fix: fold the `DELETE FROM failed_ingests` into each `persist_*` transaction** so the payload write and the failure clear commit atomically ŌĆö there is no longer an inter-transaction window to crash in. The delete runs at the **top of the transaction, before any early-return** (parse's existing-doc short-circuit, caption's `analysis_json IS NULL` no-op guard), so a unit completed via a sibling document (caption UPDATE matches nothing) still sheds a lingering ghost row. `persist_parse` already holds `parsed.file_hash`; `persist_caption` / `persist_summary` look the hash up by `image_id` / `document_id` inside the same transaction (one cheap indexed read) rather than threading a new field through `ImageCaption` or the `persist_summary` signature ŌĆö that keeps the change to `writer.py` and respects that the caption stage legitimately keys on `image_id` while the failure row keys on the image's byte-hash.

**The redundant external clears were removed, not left as no-ops.** The issue floated leaving the three `writer.clear_failure(...)` call sites in place as harmless no-op deletes "to keep this single-file." Rejected in favour of the repo's no-dormant-code rule: the three call sites are deleted, and the public `clear_failure` (now uncalled in production) becomes a private `_clear_failure` whose leading underscore *is* the contract ŌĆö **the caller must already hold the connection's transaction** (it opens none of its own, so it can compose inside `with self.conn`). Its unit test wraps it in `with conn` accordingly; per-stage clearing is now covered by `persist_*`-clears-the-row tests in `test_provenance.py`.

Additive / non-schema ŌĆö part of the v0.8.10 ingest-failure-ledger omnibus (#315), no `SCHEMA_VERSION` bump, no re-ingest.
